# Day 4 — 竞品全景：12×8 矩阵与生态定位

📌 数据截止：2026 年 3 月 18 日 | 生产方式：OpenClaw Agent + NotebookLM + 人类编辑三方协作

## 编者按

前三天，我们把 OpenClaw 拆了个底朝天。

Day 1 看骨架——Gateway+LLM+渠道的三层价值栈。 Day 2 看血管——消息路由、 Agent 编排、记忆系统的运转机制。 Day 3 动手术——从一行`docker run`到生产级部署的完整链路。三天下来，你对 OpenClaw"是什么、怎么跑"应该已经有了手感。

但产品经理不能只盯着自家后院。

今天我们拉远镜头，把 OpenClaw 放进整个 AI Agent 行业的坐标系里。这不是一篇"谁好谁坏"的横评——那种文章写出来一周就过期。我们要做的，是帮你建一张 **认知地图**：谁在做什么、为什么这样做、哪些路径已经拥挤、哪些区域还是空白。

剧透一个暗线：当你画完这张地图，会发现"多渠道接入+自主部署+持久记忆"这三个能力的 **交叉地带**，玩家出奇地少。这不是巧合，而是选择——大多数团队主动或被动地放弃了其中至少一项。

OpenClaw 恰好站在这个交叉口。是机遇，也是风险。


## 第一章 · 为什么需要一张竞品地图

### 从概念泡沫到形态分化

2023 年 3 月，AutoGPT 横空出世。一个能"自己给自己下任务"的 AI 脚本，两周拿下 10 万 Stars 。整个科技圈陷入集体亢奋：AI Agent 来了，通用人工智能还远吗？

冷静得很快。到 2023 年底，绝大多数 AutoGPT 的 fork 已经停更。原因很朴素——它跑不了真活儿。循环调用 GPT-4 烧钱如流水，幻觉导致任务频繁跑偏，没有记忆意味着每次对话从零开始。概念验证和生产工具之间，隔着一整个工程化的太平洋。

2024 年是分水岭。活下来的团队开始做选择题：服务谁？解决什么场景？愿意牺牲什么换取什么？不同的答案，长出了截然不同的产品形态。

到 2025 年 Q1，AI Agent 赛道已经分化为至少四条清晰的路径。

### 四条分化路径

**路径 A：开源框架派——给开发者造工具的工具**

代表：AutoGPT（182K Stars）、 LangChain（130K Stars）、 CrewAI（46K Stars）、 LangGraph（27K Stars）。

这群项目的用户画像高度一致：会写代码的开发者。它们提供 Agent 构建的"乐高积木"——编排逻辑、 Tool 接口、 Prompt 管理——但不管你的 Agent 最终长什么样、在哪跟用户见面。好处是灵活到极致；代价是从积木到成品，还有大量的工程活要自己干。

**路径 B：低代码平台派——让非技术人员搭 Agent**

代表：Dify（133K Stars）、 Coze（字节跳动）、 Flowise 。

思路截然不同：拖拽式画布、可视化工作流、一键发布。目标用户从开发者扩展到了运营、产品甚至市场人员。 Dify 在 2024 年的增长尤其凶猛，从年初的 20K Stars 飙升到如今的 133K 。它们解决了"不会写代码也能搭 Agent"的问题，但通常绑定特定的云服务，自部署灵活度打折扣。

**路径 C：大厂内嵌派——Agent 能力嵌入现有生态**

代表：OpenAI GPTs 、 Microsoft Copilot 、 Google Gemini Extensions 。

不另起炉灶，而是在已有的十亿级用户平台里"长出"Agent 能力。 GPTs 让 ChatGPT 用户自定义助手，Copilot 把 AI 注入 Office 全家桶。优势是分发渠道无敌——用户已经在那了；劣势是你只能在大厂划定的沙盒里玩，数据、模型、能力边界全由平台说了算。

**路径 D：消息平台原生派——在聊天界面里长出 Agent**

代表：Slack AI 、飞书智能伙伴、 Teams Copilot 。

逻辑很直接：企业沟通已经在 IM 里发生了，Agent 就应该在 IM 里服务。但它们天然只服务于自家平台的用户，跨渠道？不存在的。

### OpenClaw 的有趣之处

把四条路径摊开看，OpenClaw 不完全属于任何一条。

它是开源的（像路径 A），但不只提供框架，而是交付一个完整的"AI 助手运行时"。它支持可视化配置（靠近路径 B），但核心交互发生在用户已有的消息平台里（靠近路径 D），而不是自建一个新界面。它不依附任何大厂生态（区别于路径 C），却能同时接入 Discord 、 Telegram 、 Slack 、飞书等多个渠道。

这种"四不像"，恰恰是值得研究的地方。

### 八维对比框架

| 维度 | 核心问题 |
|---|---|
| ① 部署模型 | 纯云/自部署/混合？用户对数据有多少控制权？ |
| ② 渠道支持 | 支持哪些消息平台？能同时接入几个？ |
| ③ Agent 架构 | 单 Agent/多 Agent？线性/图状/自由循环？ |
| ④ 记忆系统 | 有没有跨会话记忆？记忆存在哪？ |
| ⑤ 安全模型 | Tool 调用有沙箱吗？权限如何控制？ |
| ⑥ 成本结构 | 平台费+Token 费的组合方式？ |
| ⑦ 开源程度 | 核心代码开源吗？许可证友好吗？ |
| ⑧ 生态成熟度 | 插件/社区/文档/第三方集成？ |

::: tip PM Takeaway
AI Agent 四条路径各自回答不同核心问题。框架派回答"怎么建"，低代码派回答"谁能建"，大厂派回答"在哪用"，IM 派回答"怎么触达"。但没有哪条路径同时回答了所有问题。

:::


## 第二章 · 开源自主 Agent 框架——理想主义者的战场

### AutoGPT：182K Stars 背后的转型阵痛

AutoGPT 可能是 AI Agent 概念最成功的"布道者"。 2023 年 3 月 30 日上线，两周突破 10 万 Stars 。截至今天，182K Stars 。

但 Stars 不是用户。

AutoGPT 的故事是一部"实验项目被迫长大"的转型剧。 2024 年起大规模重构，推出三件套：AutoGPT Server + Agent Builder + Marketplace 。从实验项目变成 **AI 工作流平台**，AI 版的 Zapier 。

但执行跟不上愿景。 Cloud Beta 在 2024 年底开放 waitlist，截至 2025 年 Q1 仍未全面开放。社区注意力转移——2024 年全年月均 commit 数量比 2023 年下降约 40%。

**跟 OpenClaw 的本质差异**：AutoGPT 想做"AI 工作流平台"（后台批处理），OpenClaw 想做"AI 助手 Gateway"（前台实时对话）。一个是 AI 版 Zapier，一个是 AI 版消息路由器。

☁️ 观测点①：AutoGPT Cloud Beta 推进速度是判断其平台化是否成功的关键指标。即使是头部开源项目，云托管的商业化路径也尚未跑通。

### CrewAI：多 Agent 编排的魅力与局限

CrewAI 的核心是 **角色扮演编排**：定义 Agent 的角色（Role）、目标（Goal）、背景故事（Backstory），组成 Crew 分配 Task 。 46K Stars，PyPI 月下载量一度突破百万。

三个明显局限：不面向实时对话（批处理模式）、无原生渠道、无持久记忆。

**但 CrewAI 跟 OpenClaw 是互补关系**：用户在 Discord 说"帮我调研东南亚电商市场"→OpenClaw 接到消息→后台启动 CrewAI Crew 跑完→OpenClaw 推回结果。前台接待用 OpenClaw，后台重活用 CrewAI 。

### LangGraph/LangChain：底层工具链

"为什么不用 LangChain 自己搭一个？"——你可以，但需要自己解决渠道接入、持久记忆、部署运维、安全边界。

LangChain 130K Stars，LangGraph 27K Stars 。 LangGraph 引入有限状态机概念，比线性 Chain 灵活得多。正在成为"认真做 Agent"的开发者首选编排工具。

但根本问题不变： **编排完了，用户在哪跟它对话？对话历史存在哪？** 这正是 OpenClaw 填的坑。

### 小节对比表

| 维度 | AutoGPT | CrewAI | LangGraph | OpenClaw |
|---|---|---|---|---|
| 核心隐喻 | AI 版 Zapier | AI 版项目组 | AI 版状态机 | AI 版消息路由器 |
| 面向用户 | 开发者→拓展中 | 开发者 | 开发者 | 开发者+高级用户 |
| 实时对话 | ❌ 批处理 | ❌ 任务执行 | ⚠️ 需自行实现 | ✅ 原生支持 |
| 渠道支持 | ❌ 无 | ❌ 无 | ❌ 无 | ✅ 多渠道统一 |
| 记忆系统 | ⚠️ 基础 | ⚠️ 基础 | ⚠️ 需自行实现 | ✅ 跨会话持久 |
| 部署复杂度 | 🔴 高 | 🟡 中 | 🟡 中 | 🟢 低 |
| Stars | 182K | 46K | 27K | — |
| 最佳场景 | 后台自动化 | 多角色批处理 | 复杂逻辑开发 | 多渠道 AI 助手 |

::: tip PM Takeaway
开源框架解决了"如何构建 Agent 内部逻辑"，但集体留下了两个问题没回答——"用户在哪跟 Agent 交互"和"Agent 怎么记住用户"。这两个"留白"，恰好是 OpenClaw 的主战场。

:::


## 第三章 · 低代码/商业 Agent 平台：人人都能造 Agent，然后呢？

2024 年最火的创业叙事之一： **"让不会写代码的人也能构建 AI Agent 。"**

逻辑很性感——大模型能力已经 commodity 化，谁能降低构建门槛，谁就拿下最大的开发者（和非开发者）市场。 Dify 、 Coze 、 n8n+AI，三条路线各自下注。

但所有人都在回答同一个问题： **怎么更容易地造 Agent？**

几乎没人在回答另一个问题： **造完之后，Agent 怎么被用起来？**

这个 gap，比大多数人想象的大得多。


### Dify：开源 LLM 应用开发平台标杆

### 一句话定位

**Backend-as-a-Service for LLM Apps 。** 你负责设计 Agent 逻辑，Dify 负责跑起来、暴露 API 。

如果说 LangChain 是给 Python 开发者的"LLM 乐高积木"，Dify 就是给更广泛团队的"LLM 应用工厂"——拖拽编排、一键部署、 API 即用。

### 核心能力拆解

Dify 走 **开源+云托管双轨模式**。 GitHub star 数突破 133k，社区活跃度在开源 LLM 项目里排前列。自部署用 Docker Compose，十几分钟能跑起来；嫌麻烦就用 Dify Cloud，注册即用。

**可视化编排** 是 Dify 最直观的卖点。工作流编辑器里，LLM 节点、条件分支、知识库检索、工具调用，全部拖拽连线。非开发者能看懂逻辑，开发者能省掉胶水代码。对于"搭个 RAG 应用"或"做个客服 bot"这类标准场景，效率确实高。

**RAG 管道** 做得相当完整。文档上传、自动分片、向量化、检索策略（混合检索、重排序），一条龙。对于企业知识库场景，Dify 把原本需要自己拼 LangChain 的活儿，打包成了开箱即用的产品。

**多模型支持** 是另一个亮点。 OpenAI 、 Anthropic 、本地 Ollama 、各种国产模型，配个 API key 就能切换。在模型层面，Dify 不绑定任何一家。

**Agent 模式** 让 Dify 不只是"RAG 工具"。你可以配置 Agent 使用 Function Calling 调用外部工具——查天气、查数据库、调 API——模型自己决定什么时候调什么工具。

### 但 Agent 在 Dify 里是什么？

这个问题值得拆开看。

Dify 里的 Agent，本质是 **一个可调用工具的 chatbot endpoint**。你在 Dify 里设计好 Agent 逻辑，Dify 给你一个 API endpoint 。前端怎么接、用户怎么触达，是你自己的事。

Dify 自带一个分享链接（Web App 形态），可以把 Agent 变成一个网页聊天窗口。但这就是它的"渠道"上限了。

想让 Agent 出现在 Slack 里？自己写集成。飞书？自己写。微信？自己写。 WhatsApp？自己写。

**Dify 产出的是 API endpoint，不是一个活在用户日常沟通工具里的"人"。**

这不是 Dify 的 bug，是它的设计选择。 Dify 定位是 **开发平台**，不是 **运行时基础设施**。它解决的是"怎么构建 LLM 应用"，不是"Agent 构建完了怎么交付到用户手里"。

### 与 OpenClaw 的维度差异

| 维度 | Dify | OpenClaw |
|------|------|----------|
| Agent 是什么 | 可调工具的 chatbot endpoint | 有记忆、有渠道、有审批的持久助手 |
| 产出物 | API endpoint + Web App | 活在聊天软件里的"人" |
| 渠道 | API（渠道自己接） | 原生多渠道 |
| 记忆 | 会话内上下文 | 跨会话持久记忆 |
| 部署目标 | 服务器/云 | 本地优先 |
| 核心用户 | 开发者/技术团队 | 想要 AI 助手的个人和小团队 |

**一个类比：** Dify 是"Agent 的 IDE"，OpenClaw 是"Agent 的操作系统"。你可以在 Dify 里设计 Agent 的大脑，但你需要 OpenClaw 这样的东西让 Agent"住"在用户的世界里。


### Coze（扣子）：字节跳动的 Agent 野心

### 一句话定位

**中国市场最大的 Agent 构建平台。** 背靠字节生态，用"零门槛+丰富插件+字节渠道"的组合拳，把 Agent 构建的门槛压到了最低。

### 核心能力拆解

Coze 的 **插件市场** 是它最大的差异化壁垒。数百个预制插件，覆盖搜索、图像生成、数据查询、各种第三方 API 。用户不需要理解 API 是什么，勾选插件就能让 Agent 获得新能力。

**与字节生态的深度集成** 是 Coze 的杀手锏。 Agent 可以直接发布到豆包 App 、飞书、甚至通过飞书机器人进入企业场景。

### 纯云模式：便利与代价

Coze 是 **纯云平台，不支持自部署**。你的 Agent 逻辑、 prompt 、知识库数据、用户对话记录—— **全部存储在字节跳动的服务器上**。

对个人玩家零运维体验丝滑。但对数据合规有要求的企业，这是一道硬墙。

**渠道能力强但封闭**：豆包 App 、飞书、微信（有限）、 Web 。想把 Agent 发到 Discord/Slack/Telegram/WhatsApp？不行。 Agent 只能生活在字节允许的渠道里。

### ☁️ 观测点②

Coze 的增长数据证明： **大量用户愿意用云托管换零部署成本。** 但纯云无自部署也引发数据焦虑。市场真正需要的是"可自部署也可云托管"的中间方案。

### 与 OpenClaw 的关键分歧

**数据主权**：Coze 在字节服务器，OpenClaw 在你本地。
**渠道锁定**：Coze 限字节生态，OpenClaw 渠道中立。
**模型锁定**：Coze 围绕豆包优化，OpenClaw 模型无关。

Coze 赌的是"用户要便利"，OpenClaw 赌的是"用户要控制权"。


### n8n + AI：工作流自动化的 Agent 化

**源码可用的工作流自动化平台（对标 Zapier），AI 能力是后来加的。**

n8n 的核心不是 AI，是"连接一切"。 400+集成连接器。 AI 节点是 2023 年后加上去的能力。

在 n8n 里， **Agent 是工作流中的一个节点** ——被调度的，不是主动的。对比 OpenClaw 里 Agent 是系统的中心。

n8n+AI 最适合： **已有 n8n 基础设施，想在工作流里加 AI 决策** 的团队。


### 低代码平台的核心洞察

**所有低代码平台都在优化"构建 Agent"，但几乎都忽略了"使用 Agent"。**

构建 ≠ 使用。低代码平台把构建门槛从"写代码"降到"拖拽"。但使用门槛呢？用户还是需要打开专门的网页/App 。 Agent 依然不在用户的日常工作流里。

**这就像造了一辆很好的车，但没有修路。**

::: tip PM Takeaway
低代码验证了"不用写代码就能用 Agent"的付费意愿。但停在了"低代码搭建"阶段，还没进入"低代码使用"阶段。真正的机会：让使用 Agent 的门槛也降到零。

:::


## 第四章 · 大模型厂商的 Agent 布局：生态红利还是生态围城？

大模型厂商自己做 Agent 是 **"房东亲自下场开店"**。它们拥有最强的模型、最大的用户基数、最深的口袋。但也面临一个结构性矛盾： **Agent 越好用→用户越依赖→越难换模型。**


### OpenAI GPTs + Assistants API

### GPTs：十亿级实验的启示

2023 年 11 月发布， **超过 1000 万个 GPTs 被创建**。但冷静看： **GPTs 本质上是"增强版 prompt 模板"，不是有记忆的持久伙伴。**

每个 GPT 就是一段系统 prompt+可选知识库+可选 Actions 。没有跨会话持久记忆，没有主动行动能力。 **更关键：GPTs 只活在 ChatGPT 界面里。** 想让它出现在 Slack/飞书/Discord？不行。

### Assistants API

比 GPTs 更灵活：Code Interpreter + File Search + Function Calling 。但局限同样清晰：模型锁定（只能用 OpenAI）、渠道锁定（自己搭桥）、记忆有限（Thread 上下文限制）、长对话成本黑洞。

GPTs 最大贡献是 **市场教育**。十亿级用户知道了"Agent"这个概念。但也暴露了"低门槛创建+高门槛使用=大量沉睡 Agent"的问题。


### Anthropic MCP：不是 Agent，而是 Agent 的"USB 接口"

**Model Context Protocol 不是 Agent 产品，是 Agent 连接外部世界的标准协议。**

MCP 定义了 LLM 怎么发现工具、理解输入输出、调用并处理结果。一个工具按 MCP 标准包装一次，所有支持 MCP 的框架都能用。它正在成为 **Agent 工具调用的事实标准**。

MCP 和 OpenClaw 是 **协议层和基础设施层的互补**：MCP 回答"Agent 能做什么"，OpenClaw 回答"Agent 在哪做"。 MCP 生态越繁荣，OpenClaw 的 Agent 能力越强。纯利好。


### Google Gemini Agent：全家桶的诱惑

**多模态 AI + Google 全家桶生态整合。** 假设你数字生活都在 Google 里，Gemini Agent 就是最优解——跨 Gmail/Calendar/Drive/Maps 理解上下文。

但 **如果你不全在 Google 里呢？** 中国用户用飞书不用 Gmail，用微信不用 Google Chat 。 Gemini Agent 的能力边界就是 Google 生态的边界。


### 大厂 Agent 的共同困局

**Agent 越好用→用户越依赖→迁移成本越高→越难换模型。**

这对厂商是护城河，对用户是锁定。今天用 GPTs 搭了 50 个 Agent，明天 Claude 出了更好的模型，你能迁移吗？不能。

**这就是"模型无关架构"的长期价值。** 当 Agent 基础设施不绑定任何一家模型厂商，你拥有的是选择权。模型能力排名每隔几个月就洗牌一次。绑定任何一家都是在赌运气。

::: tip PM Takeaway
大厂 Agent 最大优势（生态整合）=最大局限（生态锁定）。重视灵活性的用户需要"模型无关、平台无关"的 Agent 基础设施。 **你的 Agent 应该属于你，不是属于任何一家公司。**

:::


## 第五章 · 消息平台原生 Agent——离用户最近的战场

每个消息平台都在做自己的 AI 。逻辑很直接：用户已经在这里聊天了，Agent 就应该在这里服务。

但"离用户最近"和"服务用户最好"之间，隔着一堵墙。一堵叫"平台边界"的墙。


### 飞书多维表格+AI 助手

飞书的 AI 能力在 2024 年大幅增强：文档辅助写作、多维表格自动化、智能搜索、会议纪要自动生成。飞书智能伙伴（My AI）可以跨文档、表格、日程理解上下文。

但严格来说，飞书做的是 **AI 功能**，不是**AI Agent**。

区别在哪？Agent 是主动的——它可以自己决定什么时候检查你的日程、什么时候提醒你跟进任务、什么时候在群里插一句话。飞书的 AI 更多是被动的——你问它才答，你调用它才动。

更关键的限制： **飞书 AI 只在飞书生态内运作。** 你在飞书里有一个很好用的 AI 助手，但你的 Discord 社区、 Telegram 群组、 WhatsApp 家庭群——它一个都触达不了。

对于全栈飞书用户（工作沟通、文档协作、项目管理全在飞书），这不是问题。但现实是，很少有人的数字生活完全在单一平台内闭合。

### Slack AI+Agent 能力

Slack 的 AI 布局比飞书更开放一些。

**Slack AI**本身提供频道摘要、消息搜索增强、对话总结等基础能力。 2024 年推出的 **Workflow Builder AI 节点** 让用户可以在自动化工作流中嵌入 LLM 调用——比如"新消息进入#support 频道→AI 自动分类→路由到对应团队"。

更有价值的是 Slack 允许 **第三方 Agent Bot**。任何人可以通过 Slack API 构建 Agent Bot 并发布到 Slack 的 App Directory 。这意味着 OpenClaw 的 Agent 可以活在 Slack 里——但它也同时活在 Discord 、 Telegram 、飞书里。这是 OpenClaw 跟 Slack 原生 AI 的本质区别：Slack AI 只服务 Slack 用户，OpenClaw 的 Agent 跨渠道服务同一个用户。

### Teams Copilot

微软的 Agent 战略最激进，也最"重"。

Teams Copilot 不只是一个聊天助手，它横跨整个 Office 365 生态——Word 里帮你写文档、 Excel 里帮你分析数据、 PowerPoint 里帮你做 PPT 、 Teams 里帮你总结会议。背后是 Microsoft Graph 提供的企业数据访问能力。

**定价也最重：$30/用户/月。** 一个 200 人的企业，年费$72,000 。这不是试用成本，是年度 IT 预算级别的决策。但花了钱不代表用了——截至 2026 年 2 月，微软 365 的约 4.5 亿商业用户中，仅 3.3% 为 Copilot 付费，Forrester 直言"令人失望的采用率"。英国政府三个月试用 1,000 个许可证后的官方结论更直接："没有找到可靠证据表明时间节省正在转化为生产力提升。"（详见第九章踩坑录。）

Teams Copilot 的定位是"企业生产力 Agent"——假设你的企业全栈使用 Microsoft 365 。对比 OpenClaw 的"个人/小团队全渠道 Agent"，两者面向完全不同的用户群和预算级别。

### 平台原生 Agent 的结构性矛盾

把飞书、 Slack 、 Teams 摊开看，一个结构性问题浮出水面：

**每个平台都想做自己生态里的 Agent → 用户需要在 N 个平台分别维护 N 个 Agent 。**

一个真实场景：你工作用飞书，技术社区在 Discord，家人在 WhatsApp，朋友在 Telegram 。按照平台原生 Agent 的逻辑，你需要：

- 飞书里一个 AI 助手
- Discord 里一个 AI Bot
- WhatsApp 里……目前没有原生 AI 助手
- Telegram 里……也没有

四个平台，四种（或零种）AI 能力，四份（或零份）记忆，四个（或零个）上下文。

**这就是 N×N 问题**：N 个平台 × N 个 Agent = N 份割裂的体验。没有一个平台原生 Agent 能覆盖用户的完整消息版图，因为没有一个平台愿意为竞争对手的渠道提供服务。

这正是"消息路由 Gateway"架构的结构性价值： **一个 Agent，所有渠道。** 在飞书里跟它说的事，晚上在 Telegram 里它还记得。这种跨渠道的连续性，平台原生 Agent 在设计上就做不到。

::: tip PM Takeaway
消息平台原生 Agent 离用户最近，但被平台边界困死。用户需要的不是每个平台一个 AI 助手，而是一个跨平台的 AI 助手。这个需求目前几乎没有产品在认真做——除了 OpenClaw 。

:::

### 三个关键发现

**发现一：自部署与多渠道的"鱼与熊掌"困局**

看矩阵的左两列。支持自部署的产品（AutoGPT 、 LangGraph 、 Dify）普遍缺乏原生渠道支持——你得自己写飞书 Bot 、 Discord Bot 、 Telegram Bot 。支持多渠道的产品（Coze 、 Slack AI 、 Teams Copilot）普遍不支持自部署——你的数据和 Agent 都在平台手里。

自部署和多渠道，市场上的产品似乎只能二选一。

OpenClaw 是少数同时覆盖两者的产品。它自部署在你自己的机器上，同时原生支持 6+个消息渠道。这不是偶然的功能堆叠，而是"Gateway"架构设计的直接结果——Day 1 讲的"渠道层抽象"在这里体现了战略价值。

**发现二：记忆系统是重灾区**

看"记忆系统"这一列。大多数产品的"记忆"停留在 **会话级** ——对话结束，记忆消失。 GPTs 的 Thread 、 Dify 的上下文窗口、 CrewAI 的 Crew 内记忆，都是这种模式。

这意味着：你今天跟 Agent 聊了一个小时关于项目方案的讨论，明天打开新对话，它一脸茫然。

OpenClaw 的多层记忆（工作记忆/日记/MEMORY.md 长期记忆）在对比中显得异常突出。这不是因为 OpenClaw 的记忆系统有多么高深的技术——Day 2 讲过，它本质上就是文件系统——而是因为 **大多数产品根本没有认真做这件事**。

记忆是"AI 助手"和"AI 工具"的本质分界线。一个记不住你的 Agent，永远只是工具；一个记得住你的 Agent，才有可能成为助手。

**发现三：安全模型呈哑铃形分布**

两个极端：要么完全没有安全机制（大多数开源框架——AutoGPT 、 CrewAI 、 LangGraph），要么完全由平台控制（大厂产品——GPTs 、 Teams Copilot）。

中间地带几乎是空的。

OpenClaw 的"用户可配置的审批机制"——deny/allowlist/full 三档，加上 exec 审批流程——是少数占据中间地带的方案。它既不是"Agent 想干什么就干什么"的野生模式，也不是"平台替你决定 Agent 能做什么"的托管模式，而是"你自己决定 Agent 的权限边界"。

这个中间方案对 PM 来说很有参考价值：安全不应该是二选一（全开或全关），而是一个光谱——用户应该能根据自己的风险偏好调节 Agent 的自由度。

### 空白区域：谁在交叉地带？

把矩阵简化为一个二维坐标系：

**X 轴：自部署能力** （左=纯云，右=可自部署）
**Y 轴：多渠道支持** （下=无渠道或单渠道，上=多渠道）

- 左上象限（纯云+多渠道）：Coze 、 Teams Copilot 、 Slack AI
- 左下象限（纯云+少渠道）：GPTs 、 Assistants API
- 右下象限（可自部署+少渠道）：AutoGPT 、 Dify 、 LangGraph 、 CrewAI
- **右上象限（可自部署+多渠道）：几乎只有 OpenClaw**

右上象限的空旷不是因为没人想做——而是因为 **同时做好自部署和多渠道需要一个特殊的架构选择**：把渠道层从应用层彻底抽离出来。大多数产品要么把渠道做死在平台里（左上），要么根本不管渠道（右下）。

☁️ **观测点③**：这个空白区域就是 OpenClaw 云托管方案的战略机会。如果推出云托管，就能把右上象限从"技术用户专属"（需要自部署能力）扩展为"所有人可达"（注册即用），直接对标左上象限的 Coze/Slack AI——但手握"可自部署"的差异化退路。

::: tip PM Takeaway
竞品矩阵最有价值的不是看谁填满了最多格子——那只是功能清单。真正有价值的是发现 **空白区域**。当一个产品独占某个交叉区域时，要么说明这个区域没有市场需求（不太可能，考虑到数据主权焦虑和多渠道需求都在增长），要么说明这是一个被忽视的机会。对 OpenClaw 来说，这个空白可能就是它最大的故事。

:::


## 第七章 · 深层战略分析——四个关键问题

前几天（Day 1-3），我们拆了 OpenClaw 的架构、记忆系统、 Agent 编排和工具生态。功能层面的东西，聊得差不多了。

但产品经理不能只看功能。

功能是表层，架构选择才是骨架。骨架一旦长歪，后面怎么补肉都救不回来。这一章，我们跳出功能视角，问四个更深的问题。


### "做平台"还是"做基础设施"？

这是理解 OpenClaw 战略定位的第一把钥匙。

先看两条路的区别。

**平台逻辑**，Coze 和 Dify 走的路。核心动作是：建一个生态，吸引开发者进来搭 Agent，用户进来用 Agent 。流量在平台内闭环。 Coze 有 Bot Store，Dify 有模板市场。开发者来了，用户来了，网络效应起来了，平台就赢了。

这是 App Store 的逻辑。苹果不需要自己写每一个 App，它只需要让开发者愿意在这里发布、让用户愿意在这里下载。

**基础设施逻辑**，OpenClaw 走的路。核心动作完全不同：它不建生态，不搞市场，不圈用户。它提供一根管道，让你的 Agent 流向用户已经在的地方——飞书、 Discord 、 Telegram 、 WhatsApp，哪里有人，Agent 就去哪里。

这是 Twilio 的逻辑。

Twilio 是个很好的类比，值得展开说。

2008 年，如果你想让你的应用能打电话、发短信，你得跟运营商谈合同，接入 SS7 信令网络，部署 SIP 服务器。一个创业公司光通信层就能耗掉半年。

Twilio 说：别折腾了，我给你一个 API，`twilio.calls.create()`，三行代码搞定。

**关键是：Twilio 从来没有试图取代 AT&T 或中国移动。** 它不是一个新的电话网络，它是一个让开发者接入现有电话网络的中间层。

OpenClaw 做的事情高度相似。它没有试图取代飞书，没有试图干掉 Discord，没有做一个"OpenClaw Chat"让用户迁移过来。它说：你的用户在飞书？好，Agent 去飞书。用户在 Discord？好，Agent 去 Discord 。用户在微信？好，Agent 去微信。

用一句话总结：

> **Coze 是"Agent 的 App Store"，OpenClaw 是"Agent 的 Twilio"。**

对产品经理来说，这个区分至关重要。因为它决定了完全不同的竞争策略：

| 维度 | 平台（Coze/Dify） | 基础设施（OpenClaw） |
|------|-------------------|---------------------|
| 核心资产 | 用户和开发者数量 | 渠道覆盖和协议抽象 |
| 护城河 | 网络效应 | 切换成本 |
| 增长引擎 | 流量运营 | 开发者口碑 |
| 最大风险 | 用户被更大平台虹吸 | 渠道方封锁 API |

两条路没有绝对对错。但你必须选一条，因为组织能力、产品设计、商业模式全都不同。 OpenClaw 选了后者，这决定了它后面所有的取舍。


### "模型无关"是真需求还是伪命题？

"模型无关"（Model-agnostic）是 OpenClaw 反复强调的设计原则。但我见过不少 PM 对此表示怀疑：

**反方观点很直接：大多数用户只用一个模型。**

这话不假。打开你的日常工具，大概率是 ChatGPT 或 Claude 。企业客户选定一个供应商后，通常不会同时接五家。既然如此，"模型无关"是不是过度设计？是不是在解决一个不存在的问题？

如果时间是静止的，反方说得对。

但时间不是静止的。

**看一组数据：GPT-3.5→GPT-4→GPT-4o→o1→o3，这个迭代链条发生在 18 个月内。**

18 个月，五代模型。 Claude 从 2 到 3 到 3.5 到 Opus 4 。 Google 从 Gemini 1.0 到 2.5 。 DeepSeek 从零到 V3 到 R1 。

任何一个"只绑定一个模型"的系统，在这 18 个月里都至少面临一次痛苦的迁移。 Prompt 要改，输出格式要调，能力边界要重新测试，价格要重新评估。

如果你的 Agent 系统底层写死了 OpenAI 的 API 格式，突然有一天 Claude 在你的场景上表现好 50%、价格低 70%——你换不换？换，意味着改一堆代码。不换，意味着用户体验打折、成本偏高。

**所以"模型无关"的真正价值不是"今天同时用 5 个模型"，而是"明天能无痛换到更好的模型"。**

这是一份保险单。今天你可能感受不到它的价值，就像你买保险的时候觉得在浪费钱。但模型市场的剧烈变化，让这份保险的赔付概率极高。

从产品架构角度看，OpenClaw 的实现方式也值得学习。它不是简单地封装了一层 API 转发，而是在配置层面抽象了模型选择：

- 每个 Agent 可以指定自己用哪个模型
- 不同任务可以路由到不同模型（贵的模型做推理，便宜的做摘要）
- 模型切换是配置变更，不是代码变更

这意味着一个 PM 可以做这样的决策："日常对话用 DeepSeek V3（便宜），复杂推理用 Claude Opus（强），代码生成用 GPT-4o（快）。" 不同场景，不同模型，一个 Agent 搞定。

这不是过度设计。这是面向未来 18 个月的务实选择。


### 开源 Agent 的可持续性之问

聊到这里，必须直面一个尖锐问题： **开源 Agent 项目到底能不能活下来？**

AutoGPT 是一个警示。

2023 年 3 月，AutoGPT 横空出世。两周内 GitHub Stars 飙到 10 万，速度打破历史记录。媒体疯狂报道，"AGI 雏形""自主 AI"的标题满天飞。

然后呢？

Stars 增速断崖式下跌。贡献者活跃度回落。用户发现：自主 Agent 在实际场景中可靠性极低，烧 Token 速度惊人，产出质量不稳定。更致命的是，AutoGPT 缺乏清晰的商业模式和社区治理结构，核心团队维护动力持续衰减。

到 2024 年底，AutoGPT 依然存在，但已经从"改变世界"变成了"一个 GitHub 仓库"。

**OpenClaw 必须从这个案例中吸取教训。**

当前，OpenClaw 处于早期阶段。 Stars 数量、社区规模、第三方插件生态——坦率说，都还不成熟。这是它最大的风险，比任何技术问题都大。

因为开源项目的死法通常不是"技术不行"，而是"没人维护了"。

一个开源项目要活下来，必须跨越三个阶段：

**阶段一：个人项目。** 创始人凭热情和愿景驱动。代码质量可能不错，但总线因子（Bus Factor）是 1——创始人被公交车撞了，项目就死了。

**阶段二：社区项目。** 有稳定的外部贡献者，有活跃的 Issue 和 PR 讨论，有人自发写文档、做翻译、建插件。总线因子升到 3-5 。项目开始有自己的生命力。

**阶段三：可持续项目。** 有明确的治理结构（基金会或商业公司支撑），有资金来源（赞助、云托管服务、企业版），核心维护者能全职投入。想想 Linux 基金会、 Hashicorp（Terraform 背后的公司）、 Vercel（Next.js 背后的公司）。

OpenClaw 目前在阶段一到阶段二的过渡期。从 README 和社区动态看，有一定的外部关注度，但生态的飞轮还没有真正转起来。

**对 PM 的启示是：** 如果你在评估是否将 OpenClaw 纳入技术选型，"社区健康度"的权重应该不低于"功能完整度"。一个功能强大但可能两年后无人维护的框架，不如一个功能够用但生态持续增长的框架。

持续观察的指标：月活跃贡献者数、 Issue 平均响应时间、版本发布节奏、是否有企业级用户案例。这些比 Stars 数有用得多。


### "渠道统一"的长期价值

最后一个战略问题，也是我认为 OpenClaw 最被低估的能力： **渠道统一**。

先说一个直觉：

> 一个 Agent 只在一个渠道 = 一个聪明的 Chatbot 。
> 一个 Agent 在所有渠道 = 一个无处不在的助手。

这不是文字游戏，而是本质区别。

想象你搭了一个客服 Agent 。如果它只在你的网站上运行，那它就是一个网站客服——用户必须主动来找它。

但如果这个 Agent 同时在网站、微信公众号、企业微信、飞书、 WhatsApp 上运行呢？用户在哪个渠道习惯聊天，就在哪个渠道找它。这不再是"用户去找 Agent"，而是"Agent 在用户身边"。

**这里有一个类似梅特卡夫效应的动态。**

经典的梅特卡夫定律说：网络的价值与用户数的平方成正比。一个只有你一个人的电话网络毫无价值，100 万人的电话网络价值是 100 万的平方。

渠道统一存在类似的效应：

- 1 个渠道：Agent 是一个工具
- 3 个渠道：Agent 开始有存在感
- 7 个渠道：Agent 变成基础设施
- 10+个渠道：Agent 无处不在，用户甚至忘记了它是一个"产品"

**渠道数量不是线性增值，而是指数级增值。** 因为每增加一个渠道，不只是多了一个入口——而是多了这个渠道与所有已有渠道之间的协同可能。

举例：Agent 在飞书里帮你记了一个会议纪要。晚上你在 Telegram 问它"今天的会议结论是什么"，它能回答。这种跨渠道的连续性体验，单渠道 Agent 永远做不到。

这就是为什么 OpenClaw 的多渠道架构不只是一个"功能特性"，而是一个 **架构决策**。它把"渠道"从产品的固有属性变成了可插拔的配置项。

对比一下：如果你用 Coze 搭一个 Bot，它天然运行在 Coze 生态里。要接入微信？重新适配。要接入飞书？再适配一次。每个渠道都是一次开发工作。

OpenClaw 的做法是：Agent 写一次，渠道在配置文件里加。`channel: telegram`，`channel: discord`，`channel: feishu`——几行配置，Agent 就多了一条触角。

**这是基础设施逻辑的又一个体现。** 你不需要为每个渠道重新造轮子，因为 OpenClaw 的 Gateway 已经把渠道差异抹平了。


::: tip PM Takeaway
> 功能可以抄，架构选择很难改。

Coze 明天也可以加"多模型支持"，Dify 后天也可以做"多渠道适配"。但在 Platform 和 Infrastructure 之间的根本抉择，一旦做出，整个系统的数据流、权限模型、部署模式都围绕它生长。要改，等于推倒重来。

OpenClaw 的差异化不是功能级的，是架构级的。这是它最值得 PM 研究的地方——也是最容易被"功能对比表"遮蔽的地方。

:::


## 第八章 · OpenClaw 的坐标与未来战场

给 OpenClaw 画一张定位图。先说它"不是什么"：

**它不是最大的 Agent 平台。** GPTs 拥有数亿月活用户，ChatGPT 是全球下载量最高的应用之一。 OpenClaw 在用户量级上不在一个数量级。

**它不是最易用的搭建工具。** 打开 Coze，拖拖拽拽 10 分钟就能出一个能聊天的 Bot 。 Dify 的可视化编排让不会写代码的产品经理也能搭工作流。 OpenClaw 的门槛更高，需要命令行操作、编辑 YAML 配置、理解 Gateway 概念。

**它不是最强的 Agent 框架。** LangGraph 在复杂状态机编排上有更细粒度的控制。 CrewAI 在多 Agent 角色扮演上有更成熟的抽象。论单项能力的深度，OpenClaw 不是每条赛道的冠军。

**但它是唯一同时满足以下五个条件的项目：**

✅ 全开源——代码完全可审计，不存在"黑盒 Agent"
✅ 多渠道——一个 Agent，N 个触点
✅ 多层记忆——从会话记忆到长期记忆，Agent 能"记住"用户
✅ 模型无关——今天用 GPT，明天换 Claude，后天用 DeepSeek
✅ 安全审批——Agent 执行敏感操作前，先问人

你可以在市场上找到满足其中三个条件的产品，但五个同时满足？选项骤然收窄。

**这就是 OpenClaw 的生态位：交叉能力的独占区。**

不是在任何单一维度上做到最好，而是在五个维度的交集处建立阵地。这个交集目前几乎没有竞争者。

那未来呢？

如果 OpenClaw 推出云托管服务（从社区讨论看这几乎是必然方向），它的定位可以用一句话概括：

> **"可自部署的 Coze + 模型无关的 GPTs + 有记忆的 Dify"**

拆开来看：

- **可自部署的 Coze**：像 Coze 一样方便，但你可以把整个系统跑在自己的服务器上，数据不出内网
- **模型无关的 GPTs**：像 GPTs 一样强大的 Agent 能力，但不绑定 OpenAI，想用什么模型用什么模型
- **有记忆的 Dify**：像 Dify 一样灵活的工作流编排，但 Agent 能记住每一个用户的偏好和历史

这不是空想。从架构上看，OpenClaw 已经具备了实现这个愿景的骨架。缺的是产品化打磨、用户增长、以及（最关键的）一个被验证的商业模式。

未来的战场，不在"谁的功能列表更长"，而在"谁能让 Agent 真正成为用户日常生活的一部分"。能打赢这场仗的，一定是能让 Agent 无处不在、有记忆、可信任的那个系统。

OpenClaw 押了这个注。赌注够大，赛道够长。


## 第九章 · 真实用户踩坑录——当产品遇见现实

前八章的分析，大多是 PM 视角的框架拆解：谁缺渠道、谁缺记忆、谁锁定了模型。这些都是真的，但有一个维度是矩阵表格永远体现不出来的：

**真实用户在真实场景下踩的真实坑。**

这一章换一个视角——不谈架构，只谈踩坑。素材来自 OpenClaw 社区 Discord、GitHub Issues、国内技术论坛、以及龙虾生态产品用户的真实反馈（截至 2026 年 3 月）。

有些坑是产品设计缺陷，有些坑是用户预期管理失败，有些坑只是文档没写清楚——但它们都真实发生过，都值得记录。


### 外部竞品：用户实际碰到的那堵墙

#### AutoGPT：Stars 换不来可用性

AutoGPT 拿下 182K Stars，但社区里流传着一句话："**AutoGPT 是 AI Agent 的概念验证，不是生产工具。**"

真实痛点集中在三处：

**Token 失控烧钱。** 早期版本没有 Token 预算机制。用户让 AutoGPT 帮他"调研一个市场"，Agent 开始自主循环——搜索、总结、再搜索、再总结——一个下午跑下来，OpenAI 账单 $40+，任务没完成，钱没了。GitHub Issues 里这类反馈持续两年，最终在 2024 年重构中才加入预算限制，但为时已晚，用户信任已经损伤。

**幻觉导致任务跑偏不自知。** AutoGPT 的 Agent Loop 里，一旦某个推理步骤产生幻觉（比如"查到"了一个不存在的 API），后续所有步骤都会基于错误前提继续执行。更糟的是：Agent 不会自我报告"我不确定"，而是以同样的语气继续。用户在 Hacker News 上描述：让 AutoGPT 写一份竞品分析报告，拿到手后才发现里面引用的数据大半是编造的，格式看起来很专业，内容几乎不可用。

**Cloud Beta 久候不至。** 2024 年底开放 waitlist，承诺"很快全面开放"。截至 2025 年 Q1，大量用户仍在等待。社区里积累了数百条"我还没收到邀请"的帖子。有用户在 Discord 说："我已经等了四个月，已经把预算给了别家了。"

---

#### LangChain："LangSpaghetti"——开发者社区最大的爱恨情仇

LangChain 130K Stars，但它在开发者社区里触发了一场持续两年的公开讨论——规模之大、情绪之强烈，连 LangChain CEO 本人都不得不亲自下场回应。

2024 年 6 月，一篇《为什么我们不再用 LangChain 构建 AI Agent》的技术博客被提交到 Hacker News，引发 480 点赞、297 条评论的讨论风暴。几个被反复引用的原话：

> **"它很快就显而易见——LangChain 是最没用的库之一……在生产中完全没有用武之地。"**
> — @infecto，Hacker News，2024 年 6 月

> **"LangChain 应该叫 LangSpaghetti 才对。"**
> — @chatmasta，同一讨论串

> **"LangChain 在 demo 之外根本用不了。感觉连正经的日志记录都超出它的能力范围。"**
> — @tkellogg，同一讨论串

> **"一旦你想做一件稍微原创的事，就需要穿越五层抽象去改一个细节。而且你根本不理解每一步在做什么，所以出了问题等于从零开始。"**
> — @sc077y，同一讨论串

这场讨论里最耐人寻味的一句，来自 **LangChain CEO Harrison Chase 本人的回复**：
> **"LangChain 最初的版本确实做了太高层的抽象，确实把太多东西藏起来了。"**

CEO 亲口承认抽象过度——这本身就说明了问题的严重性。

**版本破坏性更新是另一个硬伤。** v0.3 发布（2024 年 9 月）的一周内，GitHub Issues 涌入大量"升级后 `ChatOpenAI` 报错"、"`create_retrieval_chain` 无法导入"、"`RunnableSequence` 接口变了"的 bug report。LangChain 每次大版本更新都会在生产代码里埋雷，且没有提供可靠的迁移路径。

**调试体验一塌糊涂。** 多层封装意味着报错信息是封装层抛出的，不是真正出错的那层。一个开发者描述："我最后把需要的功能用 80 行代码自己实现了——直接调 LLM API，没有 LangChain——效果一样好，调试简单一百倍。"

---

#### Dify：构建完美，交付断层，webhook 是重灾区

Dify 在开发者里口碑相当好——可视化编排确实降低了门槛，RAG 管道确实开箱即用。用户痛点不在构建，在**构建完之后**。

**"我的 Agent 在 Dify 里，用户在飞书里。"** 这是最高频的抱怨。Dify 给你一个 Web App 链接，但大多数企业用户的日常沟通在飞书或企业微信里。要把 Dify Agent 接入飞书，你需要自己写一个 webhook 转发服务，处理消息格式转换、Token 认证、重试逻辑……实际上是另起了一个工程项目。有用户在社区里说："我在 Dify 上花了两天搭好 Agent，然后又花了三天让它出现在飞书里——这三天完全不是 Dify 的事，是我自己在写胶水代码。"

**Webhook 系统长期不稳定。** 这不是个人遭遇，GitHub Issues 里有记录在案的连环 bug：v1.10.0 的 webhook 在 production URL 下完全无法接收内容（debug URL 正常）；v1.10.1 的 webhook 触发返回 404；v1.11.0 升级后 webhook 任务卡在"running"状态永不完成。这些 bug 横跨多个版本，说明 webhook 架构存在系统性问题，而不是偶发故障。更极端的是：Facebook/Meta 的渠道集成在架构上根本无法接入——Dify 的 HTTP trigger 只支持单一方法，而 Meta 要求同一 URL 同时处理 GET 验证和 POST 事件。这个 Issue 被关闭为 `not_planned`。

**跨会话失忆。** Dify 的上下文是会话级的。用户今天在 Web App 里跟 Agent 聊了一个小时关于产品方向的讨论，明天打开新对话，Agent 一无所知。GitHub Issues 里关于记忆问题的报告标题甚至以"(Again)"开头——暗示这是反复出现的老问题。有用户反馈：用了几周后就放弃了，"因为每次都要重新解释背景，比直接用 ChatGPT 还累"。

**自部署内存泄漏。** 自部署 Docker 版本存在一个持续多版本（v0.8 到 v0.13+）的内存泄漏 bug：api 和 worker 容器内存单调递增，直到占满约 6GB，此时 LLM 调用效率急剧下降直到瘫痪。这个 bug 已在多个环境（Docker + Kubernetes）得到验证，却长期未能彻底修复。

---

#### n8n + AI Agent：看起来在跑，其实在空转

n8n 作为工作流自动化工具广受好评，但它在 2025 年推出的 AI Agent 节点（尤其是 Tools Agent V3）在生产环境中暴露了一系列结构性缺陷——几乎每一个都经过 GitHub Issues 和社区论坛的多人复现确认。

**工具调用被静默吞掉。** 当 Claude 模型在同一次响应里同时返回文本内容和工具调用指令时（所谓"混合响应块"），n8n 的 AI Agent 节点**既不执行工具，也不报错**，直接进入下一轮 LLM 请求。从用户视角看，Agent 似乎在"思考"，实际上在空转——反复向 LLM 发请求但永远没有结果，表现酷似幻觉循环。这个问题在 n8n v2.9.4 中由社区用户提交（GitHub Issue #26352，2026 年 2 月），截至撰文时仍处于 **Open** 状态。

**V3 随机崩溃：越简单的消息越容易触发。** Tools Agent V3 的 `executeBatch.ts` 在处理简短回复（如"Ok"、"好的"）或首条消息时，会随机抛出 `Cannot read properties of undefined (reading 'reduce')` 错误——复杂的多工具调用反而不出问题。社区专家分析后发现，这是 V3 内部数据结构在特定 LLM 返回格式下为 `undefined` 所致，给出的建议是：**降回 V2.x 是目前最稳定的解决方案**。

**多 Agent 记忆串漏。** 当多个 AI Agent 共享同一个 Postgres 记忆节点时，Agent B 的输出会写入 Agent A 的记忆。一位用户描述了真实事故：Agent A 把 Agent B 生成的摘要直接发给了真实用户——因为它从共享记忆里读到了不属于自己的对话历史。n8n 目前没有"只读记忆"或"写隔离记忆"的原生支持，临时方案是用 Chat Memory Manager 节点手动隔离，但这增加了大量工程复杂度。

**生产内存泄漏。** 自托管 n8n 实例在跑 AI 工作流数周后，Node.js 堆内存持续增长不回收，最终触发 OOM（Out of Memory）崩溃。社区用户在 v1.123.19 的 Docker/SQLite 环境中记录了完整的崩溃日志：内存从正常水位单调攀升至 2GB 后，GC 完全失效，容器被 kill。社区专家确认根因之一是 AI Agent 节点在每次执行中累积上下文和历史，且没有释放机制。

---

#### Coze：便利的代价

Coze 是中国市场增长最快的 Agent 平台之一，但它的投诉也最集中在一个点：**数据去哪了？**

**数据焦虑。** 企业用户反复问同一个问题："我们的知识库上传到 Coze，字节跳动会用来训练模型吗？"Coze 官方有隐私条款，但在企业采购流程里，一句"数据存在字节服务器"就足以被安全部门一票否决。有用户描述：内部推动了三个月、通过了技术评估，最后在法务审查环节卡住了，原因就是"无法确认数据主权"。

**渠道出不了字节围城。** 用户想把 Bot 发布到 Discord 服务他的海外社区——不支持。想发布到 Telegram——不支持。想发布到 WhatsApp Business——不支持。Coze 支持的渠道几乎全是字节生态或中国市场渠道。有出海需求的团队最终不得不另起炉灶。

**开源版与云版插件生态完全隔离。** 2025 年 7 月字节开源了 Coze Studio，但自部署用户发现：开源版只附带 **18 个插件**，而云版插件市场有**成千上万个**。更关键的是，两套生态在架构上**完全隔离**——开源部署没有任何机制可以浏览、安装或代理云版插件市场的内容。与 Dify 支持从 GitHub 一键安装 `.difypkg` 插件包不同，Coze 开源版连社区分发渠道都没有。需要 OAuth 授权的插件更是完全无法使用——自部署 UI 根本没有授权配置入口。这个问题在 GitHub Issue #488 里被详细讨论，Coze 团队承认"联动已在规划中"——但没有给出时间表。选择自部署 Coze 的团队，实际上拿到的是一个被阉割了插件生态的空壳。

---

#### OpenAI GPTs：一千万个沉睡 Agent

"超过 1000 万个 GPTs 被创建"——这个数字经常被当成成功案例引用。

但另一个数字很少被提：**活跃使用的 GPTs 占比极低。**

**发现难、留存差。** GPTs 的 Store 里有数百万个 Agent，但缺乏有效的发现机制。用户创建的 GPT 除了自用，几乎没有传播路径。ChatGPT Plus 用户打开 GPT Store，面对的是一堆排名不透明的列表，找到真正好用的 GPT 基本靠口碑推荐。

**没有记忆。** 这是 GPTs 用户反馈最多的功能缺失。每次开新对话，之前的上下文全消失。用户在 Reddit 的 r/ChatGPT 子版块里有一个长期置顶的问题："为什么我的 GPT 不记得我是谁？" 回复几乎全是"这是设计如此，不是 bug"——但用户的挫败感并没有因为这个解释而消失。

**Assistants API 的成本黑洞。** 有开发者在 HN 分享：用 Assistants API 做了一个客服机器人，上线第一个月账单 $800+，原因是 Thread 上下文的存储费用被严重低估，加上每次调用都要加载完整的 Thread 历史。他最后重写了整个后端，用普通的 Chat API 替代，成本降到 $60/月。社区里更系统的分析揭示了根因：每一条消息的来回成本在 $0.07–$0.10 之间，而隐藏的 `myfiles_browser` 工具会递归浏览文件、在每次迭代中消耗完整上下文 Token，单次 Agent Run 极端情况下可达 $5–$10+。最讽刺的是：**Assistants API 没有任何内置的成本控制机制**——不设上限，不可预估，只能事后看账单。

**整个 API 从未离开 Beta，即将被弃用。** 这是最致命的一击：2025 年 8 月 26 日，OpenAI 正式宣布 Assistants API 进入弃用倒计时，**硬截止日期为 2026 年 8 月 26 日**——届时所有对 `/v1/assistants` 和 `/v1/threads` 的请求将直接停止响应。替代方案是全新的 Responses API + Conversations API，概念模型完全不同（Assistant→Prompt、Thread→Conversation、Run→responses.create()）。社区开发者的反应很能说明问题：一位正在基于 Assistants API 构建产品的开发者在论坛写道——"我一直在围绕它做产品，现在被告知有个新 API。**新 API 也是 Beta 吗？我们怎么能信任它不会再来一遍？**" 另一位开发者的评论更直白："所以 Assistants API 从头到尾就是 Beta 状态啊。" 对任何正在评估 Agent 平台的 PM 来说，这是一个教科书级的平台风险案例：**你的技术栈可以一纸公告变成废品**。

---

#### Teams Copilot：$30/月买来了什么

第五章提到 Teams Copilot 的 3.3% 付费率和英国政府"无生产力提升"结论。这里展开说说踩坑细节。

**付费率背后的市场份额暴跌。** 3.3% 的付费率（约 1500 万席位 / 4.5 亿商业用户）已经够低了，更值得注意的是趋势：在已经使用 AI 工具的用户中，Copilot 的市场份额从 2025 年 7 月的 18.8% 下降至 2026 年 1 月的 11.5%，而 Google Gemini 同期从 12.8% 升至 15.7%。微软内部也在调整预期——2025 年 12 月将 AI 销售目标削减了一半，因为不到 20% 的 Azure 销售人员能完成 Copilot 配额。Forrester 副总裁 J.P. Gownder 直言这是"令人失望的采用率"。

**英国政府试用：Excel 更慢、PPT 更差。** 英国商业贸易部（DBT）对 1,000 个许可证进行了三个月试用（2024 年 10 月至 12 月）。官方评估报告的结论不留情面：Excel 场景下 Copilot 用户比不用的人**更慢且质量更差**；PowerPoint 场景下虽快了 7 分钟，但输出质量和准确度**低于人工**，需要额外纠错。22% 的用户在试用中发现了幻觉问题。

**澳大利亚监管机构直接起诉微软。** 2025 年 10 月 27 日，澳大利亚竞争与消费者委员会（ACCC）向联邦法院起诉微软，指控其在将 Copilot 捆绑进 Microsoft 365 订阅时误导了约 270 万澳大利亚用户——自动续费用户被告知要么接受涨价（个人版涨 45%，从 AUD $109 涨到 $159/年），要么取消订阅，**故意隐瞒了第三个选项**：以原价续订不含 Copilot 的"经典版"方案。这个隐藏选项只有在用户**启动取消流程时**才会出现。ACCC 正在寻求处罚、禁令和消费者补偿。


### 龙虾生态产品：中国市场的踩坑图鉴

以下内容来自 OpenClaw 中文社区的真实集成记录和用户反馈，部分来自 Discord #cn-support 频道和 ClawHub 论坛（截至 2026 年 3 月）。

#### QClaw（腾讯）：排队盛况背后的等待焦虑

2026 年 3 月初，深圳腾讯总部出现了近千人排队安装 QClaw 的场景，"全民养虾"一度成为科技圈热梗。但繁荣背后有一个现实：**QClaw 目前仍处于邀请制内测阶段。**

普通用户的真实体验：填了申请表，等了两到四周，没有任何进展反馈。有用户在社区里晒出了自己的等待记录："3 月 6 日申请，3 月 19 日仍未收到邀请。在这两周里我已经用 Hetzner 自己把 OpenClaw 跑起来了。" 评论下面十几条"我也是"的回复。

QClaw 的核心吸引力是**微信直连**和**远程操控 PC**——这两个能力在官方 OpenClaw 上配置繁琐，QClaw 做了深度适配。但代价是：一旦进了 QClaw 的生态，切换模型的窗口就关上了。

---

#### KimiClaw（月之暗面）：踩坑记录最丰富的产品

KimiClaw 是目前龙虾生态里集成记录最详细、踩坑报告最多的产品。社区里有一份专门的踩坑文档在流传，整理了以下高频问题（来源：kimi-claw 集成 OpenClaw 踩坑记录 2026.3.14）：

**坑 1：配置路径错误导致网关拒绝启动。**
很多用户在 `openclaw.json` 的 `channels` 字段下直接添加 `kimi-claw`，结果网关启动时报错 `unknown channel id`，Agent 完全起不来。原因是 kimi-claw 是**插件渠道**，不能放在内置渠道字段里，配置方式完全不同。这个坑在文档里有说明，但位置隐蔽，新手几乎必踩。

**坑 2：Kimi + 飞书并发导致"空回复"。**
当用户同时在 Kimi App 和飞书向同一个 Agent Session 发消息时，偶发性地出现 Agent 返回 `undefined`——用户侧看到空白回复，没有任何报错提示。排查了半天才发现是并发访问同一 Session 的竞争条件（Race Condition）。这个问题在 Issue 列表里已有记录，修复在 v3.x roadmap 里，目前需要用户手动设置 Session 锁。

**坑 3：ACP Bridge 429 限流。**
重启网关时频繁触发 `bridge connection failed: 429`，尤其是短时间内多次重启后。系统会自动指数退避重连，但用户不知道这是正常行为，以为 Agent 挂了，反复重启反而加剧了 429 频率。

**坑 4：NAS Docker 环境特有问题。**
把 KimiClaw 部署在群晖 NAS 上的用户发现：NAS 的 SSH 环境默认不包含 `/usr/local/bin`，直接运行 `docker` 命令报 `command not found`，需要改用全路径。此外容器内没有 `jq` 工具，处理 JSON 配置只能用 `node -e` 替代。这两个问题在通用 Linux 环境里根本不会出现，但 NAS 用户群体量不小，社区里关于这两个问题的求助帖有几十条。

**总结**：KimiClaw 的能力上限高（集成了月之暗面的 24/7 助手生态），但对非标准环境的适配不足，文档的新手友好度有待提升。

---

#### MaxClaw（MiniMax）：10 秒部署的代价

MaxClaw 的宣传重点是"10-18 秒完成部署"和"内置 10,000+ 预配置专家 Agent"。宣传是真的——部署确实快，Agent 模板确实多。

但用了一段时间后，用户的核心抱怨集中在一件事：**模型锁死了。**

MaxClaw 的底层硬绑定 MiniMax 模型，无法配置 Fallback 链，无法切换到 Claude 或 GPT-4o。对于大多数日常任务，MiniMax 够用；但在用户遇到需要深度推理的场景（复杂代码审查、多步骤分析）时，模型天花板明显，而他们没有办法临时切换到更强的模型。

有用户在社区里写道："MaxClaw 是个好用的工具，但我感觉我在租一辆车，不在开自己的车。"

定价也是一个摩擦点：¥39/月起的订阅费，对个人开发者不算高，但它叠在已有的 MiniMax API 费用上。用户算了一笔账，发现长期使用成本其实比自己在 Hetzner 租一台机器跑原版 OpenClaw 更高。

---

#### 其他生态产品快览

**miclaw（小米）**：仅限小米 17 系列手机封测，移动端方案。不兼容 ClawHub，无法直接运行 13,700+ 官方 Skills。适合"小米全家桶"用户，出圈能力有限。

**NanoClaw**：4,000 行代码的极简版本。没有可视化 Dashboard，所有操作强制走 CLI 和 Claude 指令。对技术用户是极简主义，对普通用户是使用障碍。

**ZeroClaw**：用 Rust 重写的高性能版本。启动快、资源消耗低，但与官方 Node.js 生态存在兼容性断点——部分依赖 Node.js 的 Skills 需要额外适配。定制化开发对 TypeScript 用户不友好。

**EasyClaw**：通过自建网关代理实现"无需 API Key"的零配置体验。入门门槛极低，但高频使用时代理延迟明显，且用户无法精细调控模型底层参数。适合试用，不适合生产。


### 踩坑录的三个共同结论

把以上所有案例摊开，能看到三个反复出现的模式：

**① 构建容易，交付难。** 几乎所有平台（Dify、Coze、GPTs）都把精力放在了"让你更容易搭一个 Agent"上，但 Agent 搭完之后，怎么让它出现在用户真实使用的渠道里，几乎没人认真解决。用户被迫成为"胶水工程师"，在 Agent 和用户之间自己建桥。

**② 便利性和控制权的永恒博弈。** 每一个"更简单"的封装版（QClaw、MaxClaw、EasyClaw），都在某个维度上牺牲了控制权。模型不能换、渠道不能加、成本不能控。用户最初为了省事选了封装版，等到用量上来了，就开始觉得被套住了。

**③ 文档欠债在新手身上收利息。** KimiClaw 的踩坑记录里，至少一半的问题在文档里有解答——但位置隐蔽，或者散落在不同页面。技术用户会翻文档；非技术用户会放弃。每一份藏得深的文档，都是一批流失的用户。

::: tip PM Takeaway
竞品分析的第一步往往是功能对比，但最有价值的洞察往往藏在用户的踩坑记录里。一个功能"有"和"好用"之间，隔着一个安装成功率、一段文档、一个边缘场景的稳定性。在评估任何 Agent 产品时，不妨去翻翻它的 GitHub Issues、Discord 的 #help 频道、或者国内技术论坛的踩坑帖——那里才是最真实的产品评测。

:::


## 收尾

### PM 核心收获

经过 Day 4 的深度拆解，五条最值得带走的认知：

- **架构选择 > 功能堆叠。** OpenClaw 选择做"基础设施"而非"平台"，这个决策影响了它的一切设计取舍。评估任何技术产品，先看架构哲学，再看功能清单。

- **"模型无关"是面向未来的保险。** 在模型 18 个月迭代五代的市场里，绑死一个供应商是最大的技术债。无痛切换不是锦上添花，是生存需要。

- **渠道覆盖有网络效应。** 一个 Agent 多一个渠道，不是多 10%的价值，可能是多 50%。因为跨渠道协同创造的体验，是单渠道永远无法提供的。

- **开源的护城河在社区，不在代码。** 代码可以 fork，社区不能。评估开源项目时，月活贡献者数和 Issue 响应时间比 Stars 数更重要。

- **交叉能力的独占区是真正的壁垒。** 单项能力容易被追赶，五项能力的交集极难复制。这才是产品差异化的高阶玩法。

### Day 5 预告：安全、信任与主动服务

Agent 越强大，信任问题越尖锐。

当一个 Agent 可以读你的邮件、操作你的日历、访问你的文件——你怎么确保它不会"好心办坏事"？

Day 5，我们聊三件事：

1. **安全审批机制的设计哲学** ——Agent 想执行敏感操作，必须先过人这一关
2. **信任的阶梯** ——如何让用户从"试试看"到"放心交给它"
3. **主动服务的边界** ——Agent 应该在什么时候主动说话，什么时候闭嘴

这是整个系列最"产品"的一天。不谈架构，只谈人和 AI 之间的关系。
