今天，算账。

这是整个 7 天系列里 **最不性感但最决定生死** 的一篇。

没有炫酷的架构图，没有让人兴奋的新功能，没有"AI 改变世界"的宏大叙事。有的只是数字。一行一行、一列一列、精确到分的数字。 API 账单多少？Token 烧了多少？服务器月租几何？你花在运维上的时间折合多少钱？

为什么要专门花一整天写成本？

因为 **开源≠免费**。 Agent 不是装完就零成本运行的。开源免掉的是 license fee，但运行成本从你敲下第一条命令的那一秒就开始累积。 Day 3 你把 OpenClaw 跑起来的那一刻，你的 LLM Provider 就开始计费了。 Heartbeat 每 30 分钟跳一次，每一跳都是钱。

这个认知差是很多开发者的盲区。也是 PM 在做 Build vs Buy 决策时最容易算错的那笔账。

今天的目标：把 Day 3 那三档估算（$20/$50-65/$130-150）打开，看清每一个数字里面到底装了什么。看完之后，你会具备自己算账的能力——不是靠直觉，是靠公式。


## 第一章 · 为什么要专门算一笔账

### 一张让人沉默的账单

小张是个全栈开发者，在 Day 3 跟着我们的教程部署了 OpenClaw 。过程顺利——两小时搞定，配好飞书渠道，Agent 能对话了。他很兴奋地用了一个月。

月底，Anthropic 的 API 账单到了：**$47.23**。

小张愣住了。他的预期是$15-20——毕竟 Day 3 我们写过"保守档约$20/月"。他仔细看了账单明细：Heartbeat 每 30 分钟一跳，每天 48 次，光心跳就$15；设了 3 个 Cron 任务（晨报、日历检查、邮件汇总），每天$3.6；日常对话他用得比预期多，Agent 太好用了，每天 30+轮对话——比我们"保守档 5 轮"的假设多了 6 倍。

**$47 不是被坑了。是用超了。**

但问题是：在他用的时候，没有任何机制告诉他"你今天已经花了$1.8"。 Token 消耗是无形的——你看不到它流走，直到月底账单把所有隐形的消耗变成一个显眼的数字。

这个故事不是特例。在 OpenClaw 社区的 Discord 里，"我的 API 账单比想象中高"是排名前五的高频话题。不是因为 OpenClaw 收费——它本身完全免费——而是因为 **运行一个 AI Agent 的真实成本，天然被低估**。

### 开源的免费幻觉

让我们先把一个认知 bug 修掉。

**"开源"这个词，给人一种本能的"免费"联想。这是错的。**

更精确地说：开源免的是 **许可证费用** （license fee）。 OpenClaw 的源代码以 Apache 2.0 协议发布，你可以免费使用、修改、再发布，不需要向任何人付钱。这一层是真免费。

但运行成本从第一秒开始就是真金白银。

类比：Linux 免费。但运行一台 Linux 服务器不免费。你要付服务器的钱、电费、带宽费，还要付运维工程师的工资。 Red Hat 靠什么赚钱？不是靠 Linux 的源代码——那是免费的——而是靠"帮你把 Linux 跑稳"这项服务。 Red Hat 2023 年营收$38 亿（来源：IBM 年报），卖的全是"开源软件的运行保障"。

OpenClaw 也是一样的逻辑。软件免费，但"让软件跑起来并持续运行"不免费。

这里面包含三层成本：

**第一层：显性成本（你能在账单上看到的钱）**

- LLM API 调用费——每一次 Token 消耗都有明码标价
- 服务器/VPS 月租——跑 Gateway 需要一台机器
- 域名、 SSL 证书（如果需要的话）
- 可选的第三方服务（Embedding API 、搜索 API 等）

这层最好理解，因为有账单。

**第二层：隐性成本（你付出了但没收到账单的东西）**

- 部署时间——Day 3 首次部署 3-4 小时，这是你的人力成本
- 日常运维——更新版本、调参数、排故障，每月 4-9 小时
- 学习曲线——从"完全不懂"到"能独立运维"，投入 20-40 小时
- 调试时间——"为什么 Agent 突然不回消息了？"这种问题吃掉的注意力

这层容易被忽略，因为没有发票。但对 PM 来说，这层往往比第一层更贵——一个高级开发者的时间折合小时费率，$50-150/小时不夸张。花 4 小时调试一个配置问题，等于$200-600 的隐性支出。

**第三层：机会成本（你本可以用这些时间做什么）**

- 花在运维 OpenClaw 上的 20 小时/月，如果用来做产品开发，价值多少？
- 花在学习 OpenClaw 上的 40 小时，如果用来学另一个技能，长期回报差多少？
- 选择自部署而非云托管，省下的$30/月 vs 投入的 10+小时/月——这笔账怎么算？

这层最抽象，但对决策影响最大。因为机会成本是 **看不见的替代方案的价值** ——你永远不知道那条没走的路上有什么。

### Day 3 的三档，今天拆开看

Day 3 我们给了一张快速估算表：

| 档位 | 心跳频率 | 日均对话 | 月 TCO |
|------|---------|---------|-------|
| 保守 | 60min | 5 轮 | ~$20 |
| 中等 | 30min | 20 轮 | ~$50-65 |
| 激进 | 15min | 50+轮 | ~$130-150 |

这张表在 Day 3 的语境下够用——告诉你"大概花多少钱"。但它有三个问题：

**第一，数字是打包的。**"$20/月"里面，LLM 占多少？基础设施占多少？你不知道。更关键的是，哪些成本跟使用量相关、哪些是固定的？你也不知道。

**第二，隐性成本没算。** $20 只是显性支出。加上你的时间成本呢？如果你的时间值$50/小时，每月花 8 小时运维，那真实成本是$20 + $400 = $420 。数字瞬间变了一个量级。

**第三，模型选择这个变量被低估了。** Day 3 的表格假设全程用 Claude Sonnet 。但如果你把 Heartbeat 和简单 Cron 换成 Haiku 呢？如果你把深度推理任务单独用 Opus 呢？模型混搭策略能把 LLM 成本压缩 3-5 倍。这不是微调优——这是结构性降本。

**今天的工作就是把这三个问题全部解开。**

从第二章开始，我们逐层拆解。先从最大的成本变量——Token 经济学——入手。然后是基础设施成本。然后是时间成本。最后把三层合并，给你一个完整的 TCO 公式和优化路径。

::: tip PM Takeaway
评估任何开源产品的成本，永远要问两个问题：第一，"免费"部分到底免了哪些？第二，"不免费"的部分有没有被充分计算？大多数 Build vs Buy 的决策失误，不是因为高估了 Buy 的价格，而是因为低估了 Build 的隐性成本。 AI Agent 产品尤其如此——因为 LLM 的按量计费模型，让运行成本和使用行为高度耦合，不可能提前精确预测。

> ☁️ **观测点①**：在成本分析中，注意区分两类支出—— **规模化可摊薄的成本** 和 **随用户线性增长的成本**。服务器租金、部署时间、学习曲线是前者（10 个人用和 1 个人用，服务器费一样）。 Token 消耗是后者（每多一个活跃用户，LLM 账单等比例上涨）。云托管方案的定价难题，就藏在这两类成本的比例关系里。

:::


## 第二章 · Token 经济学：选模型就是选成本量级

### 定价基本面：Input 和 Output 不是一个价

如果你只记住 Token 经济学的一条规则，记这条： **Output Token 比 Input Token 贵 3-5 倍** （个别深度推理模型如 Gemini 2.5 Pro 可达 8 倍）。

为什么？因为生成新 Token 比读取已有 Token 的计算量大得多。 Input 是"看"——模型把你的文字读进去，做一次前向传播。 Output 是"写"——模型每生成一个 Token，都要做一次采样、评估概率分布、选择最优候选。生成 100 个 Token 意味着做了 100 次决策，每一次决策都需要完整的计算图。

这个定价差异对 Agent 场景的影响尤其显著——因为 Agent 的 **Input/Output 比例极不平衡**。

一个典型的 Agent 对话轮次：

- Input（输入）：System Prompt（~3000 tokens）+ 对话历史（~2000 tokens）+ 用户消息（~100 tokens）= **~5100 tokens**
- Output（输出）：Agent 回复（~300 tokens）= **~300 tokens**

Input 是 Output 的 17 倍。但由于 Output 单价更高（以 Sonnet 的 5x 倍率计算），单轮对话的成本比大约是 Input 77% : Output 23%。不过在月度汇总中，由于工具调用、多 Agent 编排等场景产生大量 Output token， **实际的月度成本中 Output 占比会接近 50%** （详见下文月消耗表）。

这意味着什么？意味着 **省 Token 的最大杠杆在 Input 侧** ——压缩 System Prompt 、控制上下文长度、精简对话历史。 Output 侧能优化的空间相对有限（你总不能让 Agent 少说话）。

### 主流模型定价对比表

以下是截至 2026 年 3 月主流 LLM 的 API 定价（来源：各厂商官方定价页面）。所有价格为每百万 Token（1M tokens）的美元价格：

| 模型 | 厂商 | Input 价格 ($/1M) | Output 价格 ($/1M) | Output/Input 倍率 | 定位 |
|------|------|------------------|-------------------|-----------------|------|
| Claude Haiku 3.5 | Anthropic | $0.80 | $4.00 | 5.0x | 轻量/快速 |
| Claude Sonnet 4 | Anthropic | $3.00 | $15.00 | 5.0x | 均衡/主力 |
| Claude Opus 4 | Anthropic | $15.00 | $75.00 | 5.0x | 深度推理 |
| GPT-4o | OpenAI | $2.50 | $10.00 | 4.0x | 均衡/主力 |
| GPT-4o mini | OpenAI | $0.15 | $0.60 | 4.0x | 极致性价比 |
| Gemini 2.0 Flash | Google | $0.10 | $0.40 | 4.0x | 极致性价比 |
| Gemini 2.5 Pro | Google | $1.25 | $10.00 | 8.0x | 深度推理 |
| DeepSeek V3 | DeepSeek | $0.27 | $1.10 | 4.1x | 性价比 |
| DeepSeek R1 | DeepSeek | $0.55 | $2.19 | 4.0x | 推理增强 |

几个值得注意的格局特征：

**1. 价格带接近 200 倍。** 最便宜的 Gemini Flash（Input $0.10/1M）和最贵的 Opus（Input $15.00/1M）之间，相差 150 倍。 Output 端更夸张：Flash 的$0.40 vs Opus 的$75.00，相差 187 倍。选模型不是微调成本——是选成本量级。

**2. Anthropic 的 Output 溢价最高。** 三家主要厂商里，Anthropic 统一 5x 的 IO 倍率是最高的。 OpenAI 和 Google 大多在 4x 左右。这意味着 Agent 回复越长、生成内容越多，Anthropic 的成本劣势越明显。

**3. GPT-4o mini 和 Gemini Flash 是价格破坏者。** 这两个模型的定价已经低到了"几乎不用算"的程度——Gemini Flash 的 Input $0.10/1M，意味着处理 100 万 Token（约 75 万个英文单词、一整本《哈利波特》系列）只花 0.1 美元。

**4. DeepSeek 以非对称定价搅局。** DeepSeek V3 的 Input $0.27 堪比 Flash 级别，但 Output $1.10 又比 Flash 贵了近 3 倍。它的 sweet spot 是"大量 Input 、少量 Output"的场景——比如文档分析、信息提取，恰好适合 Agent 的某些工具调用。

### 模型选择是 10x 杠杆

让我们用一个具体任务来算。

**场景**：Agent 帮你总结一封 1000 字的邮件，生成 200 字的摘要。

Token 估算：Input 约 3000 tokens（System Prompt ~2500 + 邮件内容 ~500），Output 约 300 tokens 。

| 模型 | Input 成本 | Output 成本 | 单次总成本 | 与 Haiku 的倍数 |
|------|----------|-----------|-----------|-------------|
| Gemini Flash | $0.0003 | $0.0001 | **$0.0004**| 0.08x |
| GPT-4o mini | $0.0005 | $0.0002 | **$0.0007**| 0.14x |
| DeepSeek V3 | $0.0008 | $0.0003 | **$0.0011**| 0.22x |
| Haiku 3.5 | $0.0024 | $0.0012 | **$0.0036**| — |
| Sonnet 4 | $0.0090 | $0.0045 | **$0.0135**| 3.8x |
| GPT-4o | $0.0075 | $0.0030 | **$0.0105**| 2.9x |
| Opus 4 | $0.0450 | $0.0225 | **$0.0675**| 18.8x |

**同一个任务，Gemini Flash 花$0.0004，Opus 花$0.0675 。差了 169 倍。**

更大的模型不一定更好。如果 Haiku 能以 95%的准确率完成邮件摘要，而 Opus 的准确率是 98%——那 3%的提升值不值 18.8 倍的价格？对大多数日常任务来说，答案是"不值"。

但这里有一个陷阱。

### 有效成本 = 单次成本 × 完成任务所需轮数

单纯比单次调用成本是不够的。

**一个便宜但需要重试 3 次的模型，实际上比一个贵但一次搞定的模型更贵。**

举例：你让 Agent 帮你写一段复杂的 SQL 查询。

- **Haiku**：单次$0.005 。第一次写错了，需要你指出错误，Agent 修改。来回 3 轮才搞定。有效成本 = $0.005 × 3 = **$0.015**（实际 input 还会因上下文累积更贵）。
- **Sonnet**：单次$0.018 。一轮搞定。有效成本 = **$0.018**。
- **Opus**：单次$0.09 。一轮搞定，而且 SQL 质量更高。有效成本 = **$0.09**。

从"有效成本"的角度看，Haiku 和 Sonnet 几乎打平。差距从表面的 3.6 倍缩小到了 1.2 倍。

**这就是为什么模型选择不能只看定价表。** 你需要的公式是：

```
有效成本 = 单次调用成本 × 平均完成轮数 × 上下文膨胀系数
```

"上下文膨胀系数"是什么？每多一轮对话，Agent 都要把之前的全部对话历史作为 Input 送进去。第 3 轮的 Input 是第 1 轮的 2-3 倍。这个膨胀效应在多轮对话中非常显著。

**最优策略不是"全用最便宜的"，也不是"全用最贵的"——而是按任务复杂度分层路由。** 这一点，我们在第五章会展开。

### Agent 场景的 Token 消耗四大特点

Agent 的 Token 消耗模式和普通聊天有本质区别。以下四个特点，决定了 Agent 的成本结构：

**特点一：System Prompt 税**

每一次 API 调用——不管是用户主动对话还是 Heartbeat 被动唤醒——都要带上 System Prompt 。 OpenClaw 的 System Prompt 包括：

- SOUL.md（AI 人格定义）：300-500 tokens
- AGENTS.md（工作规范）：500-2000 tokens
- 运行时上下文（工具描述、技能列表等）：500-1500 tokens
- MEMORY.md + 日志片段：300-1000 tokens

合计： **2000-5000 tokens/次**。

这是一笔"固定税"。无论你问 Agent"今天天气如何"还是让它写一份 10 页的报告，System Prompt 的消耗是一样的。 Day 3 我们提到"SOUL.md 控制在 500 字以内"，就是因为这个原因——System Prompt 越长，每次调用的"底噪"成本越高。

按 Sonnet 定价（$3/1M input），3000 tokens 的 System Prompt 每次调用花费$0.009 。看似微不足道。但如果你每天有 48 次 Heartbeat + 20 轮对话 + 3 次 Cron = 71 次调用，光 System Prompt 就是$0.64/天， **$19.2/月**。

**这笔"System Prompt 税"占轻度用户总成本的 30-40%。** 优化它的 ROI 极高。

**特点二：上下文累积——对话越长越贵**

普通聊天 APP 里，你发一条消息、收一条回复，计费只看这一轮。但 Agent 是多轮对话——第 N 轮的 Input 包含前 N-1 轮的所有内容。

Token 消耗的增长曲线是这样的：

| 对话轮次 | 本轮新增 Input | 累积总 Input | 本轮 Input 成本（Sonnet） |
|---------|-------------|------------|----------------------|
| 第 1 轮 | 3,000 (System) + 100 (用户) | 3,100 | $0.0093 |
| 第 5 轮 | 3,100 + 4×(100+300) | 4,700 | $0.0141 |
| 第 10 轮 | 3,100 + 9×400 | 6,700 | $0.0201 |
| 第 20 轮 | 3,100 + 19×400 | 10,700 | $0.0321 |
| 第 50 轮 | 3,100 + 49×400 | 22,700 | $0.0681 |

**第 50 轮的单轮成本是第 1 轮的 7.3 倍。** 这就是为什么 Day 3 实测中"主动对话"的成本占比如此高——不是因为对话轮数多，而是因为后期每一轮都在为前面所有轮次"付租金"。

实际情况比这个表格更复杂。 Agent 的回复通常比用户消息长（平均 300-500 tokens），而且工具调用的结果也会塞进上下文。一轮工具调用可能往上下文里注入 1000+ tokens 。

**控制上下文膨胀的几种方式**：
- 滑动窗口：只保留最近 N 轮对话历史
- 摘要压缩：定期让模型总结前面的对话，用摘要替代原文
- Session 切割：长任务拆成多个独立 session
- lightContext 模式：对简单任务减少加载的上下文量

**特点三：工具调用开销——每次 Tool Call 是独立请求**

Agent 不只是聊天。它会调用工具（Tool Call）——读文件、执行命令、查日历、发消息。每一次工具调用都是一个完整的 API 请求-响应循环：

```
用户：帮我查一下明天的日程
  → API Call 1：Agent决定调用calendar工具（Input: 上下文 + 用户消息）
  ← Agent回复：我需要调用get_calendar工具
  → Tool执行：调用Google Calendar API
  ← Tool返回结果：明天有3个会议...
  → API Call 2：Agent拿到工具结果，生成最终回复（Input: 上下文 + 工具结果）
  ← Agent最终回复：你明天有3个会议...
```

一次"查日程"的请求，背后是 **2 次 LLM API 调用**。如果 Agent 需要调用多个工具，比如"查日程+发邮件+创建提醒"，可能产生 4-6 次 API 调用。每一次都带着完整的 System Prompt 和上下文。

更极端的情况：Agent 在执行复杂任务时，可能进入一个"推理-调用-分析-再调用"的循环。比如你让它"分析项目目录结构"，Agent 可能会执行`ls`、`cat`几个文件、`grep`搜索——每一步都是一次 API 调用。一个看似简单的任务，实际消耗可能是普通对话的 5-10 倍。

**特点四：Heartbeat 成本——即使你不用，它也在花钱**

这是 Agent 和传统聊天工具最本质的成本区别。

ChatGPT 在你不用的时候不花钱。但 OpenClaw 的 Agent 在你不用的时候 **也在花钱** ——因为 Heartbeat 。

每 30 分钟（默认设置），Gateway 唤醒 Agent，Agent 加载完整的 System Prompt 和上下文，读取`HEARTBEAT.md`，判断有没有事要做，然后回复`HEARTBEAT_OK`或执行任务。

**即使每次都是`HEARTBEAT_OK`——即"什么都不做"——也要完成一个完整的 LLM 调用。**

让我们算一下纯 Heartbeat 的月成本：

**假设条件**：
- 心跳频率：30 分钟
- 活跃时段：每天 16 小时（08:00-24:00）
- 每次心跳 Input：~3500 tokens（System Prompt + HEARTBEAT.md + 上下文）
- 每次心跳 Output：~100 tokens（HEARTBEAT_OK + 简要判断）
- 每月 30 天

| 模型 | 每次心跳成本 | 日心跳次数 | 日成本 | **月成本（纯心跳）**|
|------|------------|-----------|-------|-------------------|
| Gemini Flash | $0.0004 | 32 | $0.013 | **$0.39**|
| GPT-4o mini | $0.0006 | 32 | $0.019 | **$0.57**|
| Haiku 3.5 | $0.0032 | 32 | $0.102 | **$3.07**|
| DeepSeek V3 | $0.0011 | 32 | $0.035 | **$1.06**|
| Sonnet 4 | $0.0120 | 32 | $0.384 | **$11.52**|
| GPT-4o | $0.0098 | 32 | $0.314 | **$9.41**|
| Opus 4 | $0.0600 | 32 | $1.920 | **$57.60**|

**仅仅是"保持 Agent 活着"这件事，Sonnet 每月花$11.52，Opus 每月$57.60 。** 你什么都没让它做——它只是每 30 分钟醒来看了一眼就继续睡觉——就花了这么多。

这就是为什么 Day 3 反复强调"心跳用便宜模型"。如果你的主力模型是 Sonnet，但把 Heartbeat 和简单 Cron 单独设成 Haiku，仅这一项改动每月省下$8.45（$11.52 → $3.07）。如果你更激进地用 Gemini Flash 跑心跳，月省$11.13 。

心跳频率的调整同样有巨大影响：

| 心跳频率 | 日心跳次数 | 月成本（Sonnet） | 月成本（Haiku） |
|---------|-----------|-----------------|----------------|
| 15min | 64 | $23.04 | $6.14 |
| 30min | 32 | $11.52 | $3.07 |
| 60min | 16 | $5.76 | $1.54 |
| 2h | 8 | $2.88 | $0.77 |

从 15 分钟调到 60 分钟，Sonnet 心跳成本从$23 降到$5.76—— **砍了 75%**。而你失去的只是"Agent 发现新邮件的延迟从 15 分钟变成 60 分钟"。对大多数个人用户来说，这笔账显然划算。

### 不同使用模式的月 Token 消耗全景

综合以上四个特点，我们可以建一个更精确的模型。以下是三种典型使用模式下，使用 Claude Sonnet 4 的月 Token 消耗和成本：

| 成本项 | 轻度用户 | 中度用户 | 重度用户 |
|-------|---------|---------|---------|
| **使用画像**| 偶尔聊天，简单任务 | 日常助手，中等复杂度 | 深度依赖，复杂工作流 |
| 心跳频率 | 60min | 30min | 15min |
| 日均对话轮数 | 5 | 20 | 50+ |
| Cron 任务数 | 1 | 3 | 5+ |
| 工具调用/天 | 5 | 20 | 60+ |
| **月 Input tokens**| ~8M | ~38M | ~120M |
| **月 Output tokens**| ~1.5M | ~7M | ~25M |
| Input 成本（Sonnet） | $24 | $114 | $360 |
| Output 成本（Sonnet） | $22.5 | $105 | $375 |
| **月 LLM 总成本**| **~$47**| **~$219**| **~$735**|
| **若全换 Haiku**| **~$5**| **~$23**| **~$78**|
| **若全换 GPT-4o mini**| **~$2.1**| **~$9.9**| **~$33**|

几个关键观察：

**1. 全程 Sonnet 的真实成本远高于 Day 3 的快速估算。** Day 3 的$50-65/月（中等档）是包含了"心跳用 Haiku"等优化假设的。如果你像小张一样全程 Sonnet 不做任何优化，中度使用的真实成本在$200+/月。

**2. 模型降级的杠杆效应惊人。** 从 Sonnet 全换 Haiku，成本直降 90%。从 Sonnet 到 GPT-4o mini，降幅高达 97%。当然，模型能力也会下降——但对日常对话和简单任务，Haiku 和 mini 完全够用。

**3. Output 成本不可忽视。** 尽管 Agent 的 IO 比是 17:1（Token 量），Output 的成本贡献却接近 50%（金额）。这意味着"让 Agent 回复简洁一些"确实能省钱——但效果不如压缩 System Prompt 和控制上下文来得大。

### Token 定价趋势：年降 50-70%，比摩尔定律还快

以上所有数字都有一个重要的时间属性： **它们在快速过时。**

LLM 的 Token 定价正在以惊人的速度下降。我们拉一下 Claude 系列的价格历史：

| 时间 | 模型 | Input 价格 ($/1M) | 与前代相比 |
|------|------|------------------|----------|
| 2024 年 3 月 | Claude 3 Sonnet | $3.00 | — |
| 2024 年 6 月 | Claude 3.5 Sonnet | $3.00 | 性能提升，价格不变 |
| 2025 年 1 月 | Claude 3.5 Sonnet (v2) | $3.00 | 继续提升，价格持平 |
| 2025 年 5 月 | Claude Sonnet 4 | $3.00 | 再次提升，价格不变 |

Anthropic 的策略是"价格锚定，性能持续拉升"——同样的价格买到越来越强的模型。虽然名义价格没变，但 **性能/美元比** 在持续飙升。

OpenAI 更激进。 GPT-4 的 Input 定价从 2023 年的$30/1M 一路跌到 GPT-4o 的$2.50/1M—— **12 倍降幅**。 GPT-4o mini 更是直接打到$0.15/1M——与 GPT-4 初始定价相比，暴跌 **200 倍**。

Google 最凶猛。 Gemini Flash 系列的定价直接把"免费 tier"拉到了每天几百万 Token 的级别。

整体趋势： **能力相当的模型，价格每年下降 50-70%。** 这比摩尔定律（每 18 个月翻倍≈年降 33%）还快。

这对成本规划意味着什么？

**1. 不要基于当前定价做长期锁定决策。** 今天花$200/月的方案，一年后可能只需要$60-80 。

**2. "等 Token 更便宜"是一个合理的策略——但不是无限期等待的理由。** 因为竞争对手也在等。先跑起来积累数据和经验，比精确优化当前成本更有长期价值。

**3. 云托管方案的定价窗口在收窄。** 如果云服务商今天定价$50/月/用户，Token 降价 50%后，这个定价的合理性就需要重新论证。除非云服务商能证明他们提供的价值主要来自"Token 之外的东西"（运维、安全、合规）。

::: tip PM Takeaway
模型选择是 AI Agent 成本结构里最大的单一变量。不是优化 10%的变量——是决定 10x 差异的变量。但选模型不能只看单价，要看"有效成本"：单次成本 × 完成任务所需轮数 × 上下文膨胀系数。最优策略是 **按任务类型路由到不同模型** ——Heartbeat 用 Flash，日常对话用 Haiku/Sonnet，复杂推理才上 Opus 。这种混合策略可以在能力几乎不损失的前提下，把 LLM 成本压到纯 Sonnet 方案的 20-30%。

:::


## 第三章 · 基础设施成本：服务器、网络与看不见的电表

### VPS / 云服务器选型

OpenClaw 的资源需求非常轻量——Node.js 运行时 + SQLite + 少量内存。 2C/2G 起步完全够用。

**主流 VPS 方案对比：**

| 方案 | 配置 | 月费 | 适合场景 |
|------|------|------|---------|
| Hetzner CAX21 | 4C/8G ARM | €7.5(~$8) | 极致性价比，欧洲节点 |
| AWS Lightsail | 2C/2G/60G SSD | $12 | 入门试用，首年优惠 |
| Vultr Regular | 2C/4G/80G | $24 | 主力推荐，全球节点 |
| DigitalOcean | 2C/4G/80G | $24 | 简单易用 |
| AWS EC2 t3.small | 2C/2G | $15-18 | 弹性需求，按需计费 |

**关键发现：服务器成本在整个 TCO 中只占 10-20%，但它是最容易感知的成本。** 用户会为每月多花$10 的服务器纠结半天，却对每月"浪费"10 小时运维时间视若无睹。

### 自托管的隐藏硬件成本

**旧电脑方案：**
- 电费：24×7 运行，~50W 功耗，约$5-10/月
- 硬件折旧：按 3 年寿命分摊，$500 设备 ≈ $14/月
- 噪音、散热、空间占用——没法量化但确实影响生活质量

**树莓派方案：**
- 硬件一次性：$80(Pi 5 8G) + $30(电源+SD 卡+外壳) ≈ $3/月(3 年分摊)
- 电费：5W 功耗，<$1/月
- 但：SD 卡可靠性堪忧、散热问题、性能瓶颈

**NAS 方案（群晖/威联通 Docker）：**
- 如果已有 NAS，边际成本接近 0
- 但 NAS 的计算性能往往较弱

### 网络与流量

API 调用流量几乎可忽略（纯 JSON，KB 级）。 Webhook 同理。

需要考虑的固定成本：
- **域名**：$10-15/年（部分渠道 webhook 需要 HTTPS 域名）
- **SSL**：Let's Encrypt 免费
- **内网穿透**：Tailscale 免费方案够用

### 基础设施成本小结

| 成本项 | VPS 方案 | 家庭自托管 | 树莓派方案 |
|--------|---------|-----------|-----------|
| 服务器/硬件 | $8-24 | $14(折旧) | $3(折旧) |
| 电费 | 含在 VPS | $5-10 | <$1 |
| 域名+SSL | $1 | $1 | $1 |
| 网络 | 含在 VPS | $0(Tailscale) | $0 |
| **小计**| **$9-25**| **$20-25**| **$4-5**|

::: tip PM Takeaway
基础设施是 Agent 成本中"最小但最显眼"的部分。优化它的性价比有限——真正的成本大头在别处。

:::


## 第四章 · 隐性成本：你的时间才是最贵的资源

### 部署时间

Day 3 实测数据：首次完整部署约 3-4 小时（含环境准备、配置、调试）。

按时薪折算（假设$50/小时）：
- 一次性部署成本：$150-200
- 分摊到 12 个月：$12-17/月

这不是一次性的——每次重大版本更新、迁移服务器都会重新产生。

### 日常运维（月均 4-9 小时）

**常规维护（2-4 小时）：**
- 系统更新与安全补丁
- OpenClaw 版本升级（git pull + 测试）
- 日志检查与清理
- SSL 证书续期

**排错（1-3 小时，波动大）：**
- API 密钥过期或额度用完
- 渠道 webhook 断连
- 内存泄漏或进程崩溃
- 依赖包冲突

**调教优化（1-2 小时）：**
- Prompt 优化（SOUL.md 、技能文件）
- 测试新模型效果
- 调整心跳频率

### 学习曲线

前置知识：Linux 基础、 Docker/Node.js 、 API 密钥管理、网络（DNS/HTTPS/webhook）、 Git 。

如果上述知识为零，学习成本约 20-40 小时。这些知识有通用价值，但如果 **只是为了跑一个 Agent**，ROI 值得商榷。

### 时间成本货币化（按$50/小时）

| 成本项 | 首年月均 | 稳定期月均 |
|--------|---------|-----------|
| 部署(分摊) | $15-30 | $0 |
| 学习(分摊) | $40-80 | $0 |
| 日常运维 | $200-300 | $100-200 |
| 排错调试 | $100-200 | $50-100 |
| 调教优化 | $100-150 | $50 |
| **月均合计**| **$455-760**| **$200-350**|

**核心洞察：即使在稳定期，时间成本仍然是服务器+API 费用的 2-5 倍。**

这解释了 SaaS 模式为什么能存在—— **你以为在为软件付费，其实在为"不用操心"付费。**

类比：自己换机油 vs 去 4S 店。配件费只是小头，你的时间和舒适度才是大头。

### 反论

公平起见，自部署的时间不全是"成本"：学到的技能有长期价值，完全掌控意味着更深度定制，技术爱好者折腾本身就是乐趣。

**但对于"只想用 Agent"的用户，每一分钟运维时间都是纯成本。**

::: tip PM Takeaway
评估 Agent 产品时，把你的时间按时薪折算后加入 TCO 。如果结果让你吃惊——恭喜，你刚发现了云托管的定价空间。


:::


## 第五章 · 三档 TCO 深度拆解：你到底在为什么买单？

前面四章拆的是零件——Token 、服务器、运维时间。但没人买零件，大家买的是"完整体验"。

就像买车，你不会分别计算发动机多少钱、轮胎多少钱、保养多少钱。你问的是： **我每个月养这台车，到底花多少？**

这就是 TCO（Total Cost of Ownership，总拥有成本）要回答的问题。

为了让这个分析真正有用，我设定了三档典型用户画像。不是拍脑袋，而是基于我们在社区里观察到的三种最常见使用模式。你可以对号入座，也可以在三档之间做插值。

重要声明：以下所有数字基于 2026 年 3 月的定价水平。 Token 价格每年跌 50-70%，一年后这些绝对值大概率会变——但 **结构性结论不会变**。


### 📊 档位一：轻度用户——"个人助手"

#### 用户画像

叫他小张。独立开发者，28 岁，杭州。

小张用 OpenClaw 干什么？每天让 Agent 帮他查查文档、翻译几段英文、写点 commit message 、偶尔头脑风暴一下产品创意。日均交互 10-20 次，主要用 Haiku（因为便宜且快），偶尔复杂问题切 Sonnet 。

一句话总结： **把 Agent 当成一个永远在线的实习生**。

#### 显性成本明细

| 成本项 | 月费 | 备注 |
|--------|------|------|
| **API 调用**| | |
| ├ Haiku（日均 15 次×30 天） | $3.4 | ~450 次/月，平均 1.5K tokens/次 |
| ├ Sonnet（日均 3 次×30 天） | $2.2 | ~90 次/月，平均 2K tokens/次，Sonnet $3/$15 per MTok |
| ├ Heartbeat（4 次/天） | $0.4 | Haiku 处理，低 token 消耗 |
| **API 小计**| **$6.0**| |
| **基础设施**| | |
| ├ VPS（Hetzner CX22） | $4.5 | 2vCPU/4GB，德国节点 |
| ├ 域名 | $1.0 | 年付$12 分摊 |
| **基础设施小计**| **$5.5**| |
| **显性成本合计**| **$11.5/月**| |

看到这个数字，你可能觉得"挺便宜的啊，一杯奶茶钱"。

别急。

#### 隐性成本明细

| 时间投入 | 月均小时 | 货币化（$50/h） | 说明 |
|----------|----------|------------------|------|
| 日常运维 | 2h | $100 | 更新版本、检查日志、处理告警 |
| Prompt 调教 | 1h | $50 | 优化 SOUL.md 、调整 Skill 、微调行为 |
| **隐性成本合计**| **3h**| **$150/月**| |

::: warning ⚡ PM 旁注
时薪$50 是美国自由职业者中位数。如果你用国内开发者薪资换算（月薪 20K≈$115/天≈$14/h），隐性成本会降到~$42/月。但这不改变结构性结论——隐性成本依然远大于显性。
:::

#### TCO 汇总

```
月TCO = $11.5（显性）+ $150（隐性）= $161.5 ≈ $162/月

显性占比：7.1%
隐性占比：92.9%
```

**关键洞察**：小张每个月真正"花出去"的$11.5，只是冰山一角。他以为自己在省钱（"才 11 块多"），实际上他每个月在为这 11 块的服务投入 150 块的时间。

ROI 角度：如果一个云托管服务卖$30/月，完全省掉这 3 小时运维，小张其实"赚了"$120 。但小张多半不这么想——因为他觉得"我反正闲着也是闲着"。

这就是自部署最大的认知偏差： **人倾向于把自己的时间估值为零**。


### 📊 档位二：中度用户——"生产力工具"

#### 用户画像

叫她 Lisa 。某 B2B SaaS 公司的产品经理，32 岁，北京。

Lisa 把 OpenClaw 深度嵌入工作流：每天用 Agent 整理会议纪要、分析竞品、起草 PRD 、做数据分析、管理日程。日均 50-100 次交互，Sonnet 为主力模型（需要高质量输出），复杂分析任务用 Opus，简单查询降级 Haiku 。

一句话总结： **Agent 已经是她生产力的核心组件，不能挂**。

#### 显性成本明细

| 成本项 | 月费 | 备注 |
|--------|------|------|

$79/月，比一个 ChatGPT Plus + Claude Pro 组合（$40）贵一倍。但 Lisa 觉得值——因为 OpenClaw 能做的事远超两者之和：多渠道接入、自定义 Skill 、主动心跳、完整记忆。

问题出在下面。

#### 隐性成本明细

| 时间投入 | 月均小时 | 货币化（$50/h） | 说明 |
|----------|----------|------------------|------|
| 日常运维 | 3h | $150 | 版本更新+监控+日志审查+备份验证 |
| 排错 | 1.5h | $75 | 模型超时、 Skill 冲突、 webhook 断连 |
| Prompt 调教 | 1h | $50 | 优化输出质量、调整降级策略 |
| **隐性成本合计**| **5.5h**| **$275/月**| |

Lisa 不是开发者，很多问题她搞不定，需要找技术朋友帮忙或者去 Discord 社区提问。这部分时间更痛苦——因为她的核心价值应该在产品工作上，而不是在 debug YAML 配置。

#### TCO 汇总

```
月TCO = $79（显性）+ $275（隐性）= $354/月

显性占比：22.3%
隐性占比：77.7%
```

**关键洞察**：Lisa 的显性成本占比提升到了 22%——因为 API 用量上去了。但隐性成本依然占大头。而且 Lisa 的 5.5 小时运维，每小时都比小张更"贵"——不是因为时薪不同，而是因为 **机会成本不同**。小张的运维时间可能是周末折腾的乐趣，Lisa 的运维时间是从产品工作中偷出来的。

::: warning ⚡ PM 旁注
Lisa 是云托管服务的 **完美目标客户**。她有明确的支付能力（$79 已经在付了），有强烈的痛点（运维不是她的专长），有清晰的 ROI 对比（$275 隐性成本可以被$30-50 的托管费用替代）。如果你在做 OpenClaw 云托管产品，Lisa 就是你的 Day 1 用户。
:::


### 📊 档位三：重度用户——"自动化引擎"

#### 用户画像

叫他们 Team Alpha 。一个 5 人创业技术团队，用 OpenClaw 作为内部 AI 中台。

Team Alpha 怎么用？多 Agent 协作——一个 Agent 管代码 review，一个管客服，一个管数据分析，一个管内容生成。混合模型策略：核心推理用 Opus，日常交互用 Sonnet，批量任务用 Haiku 。每天总交互量 200-500 次，加上大量自动化 Cron 任务和 webhook 触发。

一句话总结： **OpenClaw 是他们的技术基础设施，像数据库一样不可或缺**。

#### 隐性成本明细

| 时间投入 | 月均小时 | 货币化（$50/h） | 说明 |

注意这里没有"Prompt 调教"——因为 Team Alpha 有专人负责，这个时间算在了日常运维里。但他们的排错复杂度远高于前两档：多 Agent 之间的优先级冲突、共享记忆的一致性问题、自动化任务的连锁失败……每一个都可能花掉一整个下午。

#### TCO 汇总

```
月TCO = $186（显性）+ $450（隐性）= $636/月

显性占比：29.5%
隐性占比：70.5%
```

**关键洞察**：即使到了重度使用场景，隐性成本占比依然超过 70%。而且这还是乐观估计——如果遇到一次严重故障（比如 Agent 在客户对话中发送了错误信息），修复+善后的时间成本可能是整个月运维时间的数倍。


### 三档全景对比

把三档放在一起看，模式就清楚了：

| 维度 | 轻度（个人助手） | 中度（生产力工具） | 重度（自动化引擎） |
|------|------------------|--------------------|--------------------|
| **月 TCO**| ~$162 | ~$354 | ~$630 |
| **显性成本**| $11.5（7%） | $79（22%） | $186（30%） |
| **隐性成本**| $150（93%） | $275（78%） | $450（70%） |
| **API 占比（显性中）**| 52% | 67% | 70% |
| **基础设施占比（显性中）**| 48% | 33% | 30% |
| **月运维时间**| 3h | 5.5h | 9h |
| **主力模型**| Haiku | Sonnet | 混合 |
| **痛点重心**| 折腾感 | 运维焦虑 | 系统复杂度 |

几个结论立刻跳出来：

**1. 隐性成本在所有档位都占主导**

从 93%到 70%，虽然比例递减，但绝对值递增（$150→$275→$450）。用量越大，显性成本增速更快（API 调用线性增长），但隐性成本也在增长——而且增长的是那些最令人痛苦、最难预测的部分（排错、跨 Agent 协调）。

**2. 越往上走，API 占比越高**

轻度用户的成本被基础设施"稀释"了（服务器是固定成本）。重度用户 API 已经占显性成本的 70%。这意味着 **模型选择** 对高用量场景的影响最大。

**3. 运维时间不是线性增长——是超线性**

从 3h 到 5.5h 到 9h 。用量增加了 15-25 倍（15 次→75 次→200 次），运维时间只增加了 3 倍。但复杂度的增长是指数级的——多 Agent 的调试难度远不是单 Agent 的简单倍数。


### 成本敏感度排序：什么最值得优化？

如果你只有一个下午来降低 TCO，应该从哪下手？

按影响力从大到小排序：

| 排名 | 成本因素 | 影响倍数 | 说明 |
|------|----------|----------|------|
| **1**| 模型选择 | **10-20x**| Haiku vs Opus 单次成本差 15 倍。这是杠杆最大的变量。 |
| **2**| 交互频率 | **线性**| 日均 10 次→100 次，API 成本直接 10x 。但这通常是需求驱动，不太好"优化"。 |
| **3**| 心跳频率 | **2-5x**| 从每 15 分钟→每 2 小时，心跳成本降 6x 。很多人没意识到心跳在偷偷烧钱。 |
| **4**| 服务器选型 | **2-3x**| AWS→Hetzner，同配置 2-3 倍价差。但迁移有一次性成本。 |
| **5**| 时薪假设 | **主观**| 这个不是你能"优化"的——它更多反映你对自己时间的估值。 |

::: warning ⚡ PM 旁注
注意排名第 2 的"交互频率"——它是线性影响，但通常不可压缩。你不会因为贵就少问 AI 。这就像你不会因为流量费贵就少用手机。真正可操作的优化空间在第 1 、 3 、 4 。
:::

#### 优化优先级（行动指南）

**Step 1：先选对模型（影响最大）**

90%的日常交互用 Haiku 就够了。把 Sonnet 和 Opus 留给真正需要的场景。很多人默认 Sonnet"因为输出质量好"，但实际上 Haiku 处理简单查询、翻译、格式化的效果几乎一样——而价格是 1/12 。

具体操作：在 Agent 配置中设置智能路由——根据任务复杂度自动选择模型。 Day 1 中讲的 Fallback Chain（降级链）不仅是可用性保障，也是成本控制利器。

**Step 2：控制心跳频率（最容易忽视）**

默认每 30 分钟一次心跳，一天 48 次。改成每 2 小时一次，变成 12 次——成本降 4 倍。而且大多数场景下，你真的不需要 Agent 每 30 分钟检查一次邮箱。

如果确实需要高频检查，把心跳模型降级为 Haiku 。心跳任务通常很简单（检查有没有新消息），不需要动用 Sonnet 。

**Step 3：优化 Cron 和自动化（隐性烧钱大户）**

每个 Cron 任务都在消耗 Token 。审查一下你的定时任务列表——有没有"当时调试用的，后来忘了关"的 Cron？有没有频率过高的定时摘要？

经验法则：每增加一个 Cron 任务，月成本增加$0.5-5（取决于模型和频率）。 10 个不必要的 Cron，一个月就是$5-50 的浪费。

**Step 4：最后才考虑换服务器（迁移有成本）**

从 AWS EC2 换到 Hetzner 能省 50-60%，但迁移过程本身要花 4-8 小时。如果你的显性成本本来就不高（轻度用户$11.5），省下的$3-4 不值得花一天去迁移。重度用户可以认真考虑。

#### 给每个读者的 TCO 速算公式

不想查表？用这个公式快速估算：

```
月TCO ≈ (日均调用次数 × 单次平均成本 × 30) 
       + 服务器月费 
       + (月运维小时 × 你的时薪)
```

**代入方法：**
- 单次平均成本：Haiku ~$0.005，Sonnet ~$0.02，Opus ~$0.09，混合 ~$0.015
- 服务器月费：入门$5，中配$15，高配$30-50
- 月运维小时：新手 8-12h，熟手 3-6h，团队 6-15h
- 你的时薪：自行估算（但请诚实——不要填 0）

**快速估算示例**：
- 日均 30 次 × $0.02（Sonnet）× 30 = API $18
- 服务器 $10
- 运维 4h × $30/h = $120
- **月 TCO ≈ $148**

你会发现，无论怎么调参数， **运维时间那一项永远占大头**。除非你把自己的时薪填成 0——但那不是"免费"，那是"自欺"。


## 第六章 · 假如做云托管，怎么定价？——AI 服务的定价心理学

搞清楚了成本结构，接下来的问题自然而然： **如果有人把这些成本打包成一个云服务卖给你，应该卖多少钱？**

或者反过来问——如果你是产品经理，要设计一个 OpenClaw 云托管服务的定价方案，你该怎么定？

这个问题看似简单，实际上触及了 AI 服务定价中最微妙的心理学地带。让我们先建立参照系。


### 市场定价参照系：别人在收多少钱？

在设计自己的定价之前，先看看市场上的"锚点"。

:::

#### 定价心理带

从上面的数据，我们可以提炼出 AI 服务定价的三个心理区间：

**$20/月以下——"消费者区间"**

这是冲动消费的安全区。用户不需要写报销单，不需要审批，自己信用卡刷了就行。 ChatGPT 和 Claude 锚定在这个价位，不是因为他们的成本只值$20——而是因为他们需要海量用户基数。

**$50-100/月——"专业区间"**

用户开始认真评估 ROI 。"这个工具每月$80，能不能帮我省出$80 以上的价值？"这个区间的产品需要明确的生产力提升叙事。 Dify Professional ($59)就在这里。

**$150+/月——"企业区间"**

个人不太可能自掏腰包。需要公司付费、需要发票、需要采购流程。但一旦进入这个区间，价格敏感度反而下降——$150 和$200 对采购流程来说没区别，但$19 和$150 有天壤之别（一个不需要审批，一个需要）。

**OpenClaw 云托管应该落在哪个区间？**

这取决于你要服务谁。但有一个基本判断：

> 如果你想获取最大量的用户，定在$20 以下。
> 如果你想获取最高质量的用户，定在$50-100 。
> 如果你想获取企业客户，准备好做定制报价。

大多数 OpenClaw 云托管产品（QClaw 、 KimiClaw 等）最终会落在 **$20-50 的中间地带** ——高于消费者冲动，低于企业门槛。这是一个尴尬但真实的位置。


### 三种定价模型：每种都有人在做

在确定了价格区间之后，下一个问题是定价结构。市场上已经验证过三种模型，各有优劣。

#### 模型 A：固定订阅——"Netflix 模式"

```
Starter:  $15-20/月  → 基础Agent，Haiku为主，有限交互
Pro:      $40-50/月  → 全模型接入，高频交互，心跳+Cron
Team:     $100-150/月 → 多Agent，协作，管理后台
```

**优势：**
- 用户心智最简单——"每月花这么多，什么都能用"
- 收入可预测，利于财务规划
- 毛利空间清晰（你知道每个用户的成本上限在哪）

**劣势：**
- 重度用户补贴问题——一个日均 200 次交互的用户和一个日均 10 次的用户付一样的钱
- 需要设置限额（"每月 5000 条消息"），否则被薅穿
- 限额设计是门艺术——太低用户骂，太高公司亏

**适用场景：** 面向消费者和小团队。大多数竞品选这个模型。

**关键数学**：假设 Pro 档$45/月，平均用户日均 40 次交互。
- API 成本：40×$0.02×30 = $24
- 基础设施分摊：$5/用户/月
- 运营成本分摊：$5/用户/月
- 毛利：$45 - $34 = $11，毛利率 24%

这个毛利率比 SaaS 行业标准的 70%低得多。原因是 **AI 产品的边际成本不像传统 SaaS 趋近于零——每多一个用户，API 成本是实打实的增量**。这是 AI SaaS 和传统 SaaS 最根本的区别。

#### 模型 B：按量计费——"水电模式"

```
基础费：$5/月（包含平台使用权+基础设施）
消息费：$0.01-0.03/条（根据模型档位浮动）
  - Haiku消息：$0.01
  - Sonnet消息：$0.02
  - Opus消息：$0.05
```

**优势：**
- 公平——用多少付多少
- 不存在"重度用户补贴轻度用户"的问题
- 轻度用户进入门槛极低（$5 基础+$3 消息 = $8/月）

**劣势：**
- 账单不可预测，用户焦虑（"这个月我是不是用太多了？"）
- 使用抑制效应——用户会因为"每条消息都在花钱"而减少使用，伤害活跃度
- 对话体验被撕裂——你希望用户自然地和 Agent 交流，而不是每次开口前心算"这条消息值不值$0.02"

**适用场景：** 面向 API 开发者，或者作为企业方案的计费底层。

::: warning ⚡ PM 旁注
按量计费在 AI 产品上有一个特别的毒性——它让用户把 Agent 当"工具"而不是"助手"。你会因为每条短信收费而少给秘书发消息吗？如果会，那不是一个好的"秘书"体验。 OpenClaw 的核心价值是"长期 AI 伙伴"，按量计费和这个叙事天然矛盾。
:::

#### 模型 C：Vercel 模式——"开源免费+托管收费"

```
开源版：免费（自己部署，自己运维）
托管版：$30-50/月（我们帮你部署、运维、更新）
```

核心卖点不是功能，是 **"不用操心"**。

**优势：**
- 完美对齐开源社区预期——代码永远免费
- 营销叙事强大——"你买的不是软件，是时间"
- 定价逻辑清晰——你的竞争对手是用户的时间，不是其他产品

**劣势：**
- 只能吸引"自己部署过但受不了"的用户（市场较小）
- 需要用户先体验过自部署的痛苦，才能理解托管的价值
- 利润空间取决于运维效率——如果你的运维成本和用户自己搞差不多，那利润就很薄

**适用场景：** 已经有大量自部署用户的生态。 OpenClaw 正好在这个位置。


### "不用操心"的定价数学

Vercel 模式之所以迷人，是因为它的数学特别优美。让我们算一笔账：

#### 用户自部署的"真实成本"

以中度用户 Lisa 为例：

```
显性成本：$79/月
隐性成本：$275/月
总TCO：    $354/月
```

#### 云托管服务的定价

```
托管费：$40/月（包含运维、更新、监控、备份）
API费：用户自带Key（pass-through，不加价）
总费用：$40 + 用户自己的API费用
```

或者更简单的方案：

```
全包价：$80/月（含基础API额度+运维+基础设施）
超额API：按量加价
```

#### 用户的"储蓄"

```
自部署TCO：  $354/月
云托管费用：  $80/月（全包）
月节省：      $274/月

其中：
  - 消除运维时间：5.5h/月 → 0h/月
  - 消除排错焦虑：无价
  - 获得专业级可用性：99.5%+ SLA
```

用户每月"赚"$274 。年化"赚"$3,288 。

如果你能清楚地传达这个数学，定价$80/月完全不是问题——这在用户的 TCO 里只占 23%，比自部署省了 77%。

但有一个问题——

**现实中，这个"数学"说服力远没有你想象的强。**


### 最大的营销挑战：很多人觉得自己的时间"不值钱"

上面的计算有一个致命假设：用户愿意给自己的时间标价。

现实是大量目标用户——特别是开发者和技术爱好者——有一种根深蒂固的信念：

> "我自己能搞定的事情，为什么要花钱让别人做？"

这不是理性计算的结果，而是身份认同。他们的自我价值有一部分来自于"我能自己搞定一切"。告诉他们"你的时间值$50/小时，所以你该买我的服务"，他们的反应不是"有道理"，而是"你在教我做事？"

这就是为什么 Vercel 的成功不是因为它"算账算得好"，而是因为它在开发者社区建立了一种新的共识： **部署不是你应该花时间的事情。你的时间应该花在写代码上。**

OpenClaw 云托管面临同样的挑战：你需要让用户相信， **运维 AI Agent 不是你应该花时间的事情。你的时间应该花在使用 Agent 上。**

这是一个叙事转变，不是一个数学证明。

具体怎么做？几个策略：

1. **用案例而不是算账** ——"Lisa 在用了云托管后，每周省出 5 小时做产品工作"比"你的时间值$50/h"更有说服力
2. **先让用户体验痛苦** ——免费试用自部署版，等他们遇到第一次凌晨 3 点的 Agent 宕机之后再推荐托管版
3. **把"省时间"翻译成"做更多"** ——不是"你省了 5 小时"，而是"你多了 5 小时可以用来做产品/写代码/陪家人"
4. **社区口碑** ——让已经切换的用户在 Discord/Twitter 说"我再也不想自己运维了"


### ☁️ 观测点②：BYOK vs 统一 Key——一个看似技术的商业决策

在云托管定价中，有一个决策看起来很技术，实际上深刻影响商业模式：

**API Key 谁来提供？**

两种模式：

#### 模式 A：BYOK（Bring Your Own Key）——用户自带 API Key

```
用户自己在Anthropic/OpenAI注册账号，获取API Key
输入到云托管平台
平台作为代理转发请求
API费用在用户自己的账户上结算
```

**优势：**
- 透明——用户清楚每一分钱花在哪
- 信任——"你不碰我的 Key"
- 利润结构清晰——平台只收托管费，不碰 API 费
- 用户有选择权——随时换模型提供商

**劣势：**
- **利润薄** ——平台只赚托管费$20-40/月，没有 API 加价空间
- **用户体验差** ——需要用户自己去注册 Anthropic 账号、管理额度、处理账单
- **支持复杂** ——"我的 Key 报错了"是平台的问题还是用户的问题？边界模糊
- **流失风险** ——用户有了 Key 就可以随时回去自部署

**利润数学**：
```
托管收入：$30/月
基础设施成本：$8/用户/月
运营成本：$7/用户/月
毛利：$15/用户/月（毛利率50%）
```

看起来还行——50%毛利。但考虑到获客成本（CAC）和流失率，这个模型需要非常高的用户留存才能盈利。

#### 模式 B：统一 Key——平台提供 API

```
平台和Anthropic/OpenAI签大客户协议
获得批量折扣（通常8-15%折扣）
以打包价提供给终端用户
用户不需要管API Key
```

**优势：**
- **用户体验极好** ——开箱即用，不需要管理 API 账号
- **利润空间大** ——批量采购折扣+加价空间（通常加 15-30%）
- **粘性高** ——用户没有自己的 Key，切换成本高
- **upsell 清晰** ——"升级到 Opus 套餐"比"请自行升级 API 额度"自然得多

**劣势：**
- **信任问题** ——"你会不会偷偷多收我钱？"
- **Key 安全责任** ——一旦 Key 泄露，影响所有用户
- **资金压力** ——需要垫付 API 费用（用户月底付费，但 API 调用实时计费）
- **厂商锁定** ——如果 Anthropic 涨价，你的利润直接被吃

**利润数学**：
```
包年套餐收入：$80/月
├ API成本（批量折扣后）：$40/月
├ 基础设施成本：$8/月
├ 运营成本：$7/月
毛利：$25/月（毛利率31%）
```

等等，这个毛利率怎么比 BYOK 还低？因为你承担了 API 成本。但绝对毛利从$15 涨到了$25——而且更重要的是，统一 Key 模式的 **留存率** 通常比 BYOK 高 30-50%。算上留存优势，LTV（用户生命周期价值）显著更高。

#### 实操中的第三种选择：混合模式

```
基础版：统一Key，$45/月全包（简单省心）
高级版：BYOK，$20/月托管费（自己控制成本）
```

这是目前市场上最聪明的做法——用统一 Key 版本降低入门门槛、获取大量用户，用 BYOK 版本留住那些"我要控制一切"的高级用户。

**但有一个陷阱**：BYOK 用户往往是最高价值的用户（技术能力强、使用频率高），却只付最低的费用。你可能面临"最赚钱的用户给你贡献最少利润"的窘境。

#### BYOK vs 统一 Key 决策矩阵

| 维度 | BYOK | 统一 Key | 混合 |
|------|------|---------|------|
| **用户体验**| ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **平台毛利**| 中（纯托管费） | 高（含 API 加价） | 高（看比例） |
| **用户信任**| ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **流失抵抗**| 低 | 高 | 中 |
| **实施复杂度**| 低 | 中（需签大客户协议） | 高（两套系统） |
| **资金压力**| 无 | 高（垫付 API 费） | 中 |
| **适合阶段**| MVP/早期验证 | 规模化增长 | 成熟期 |

::: warning ⚡ PM 旁注
如果你是早期创业团队，建议先从 BYOK 起步。原因很简单：不需要和 Anthropic/OpenAI 谈大客户协议（你没那个量），不需要垫付 API 费用（你没那个钱），而且 BYOK 天然建立信任（"我们不碰你的 Key"）。等用户量到了 1000+，再考虑引入统一 Key 选项。
:::


### PM Takeaway：定价不是数学题，是心理学题

回顾这一章的核心要点：

**1. $20 是心理锚点，但 OpenClaw 云托管不应该只卖$20**

ChatGPT 和 Claude 的$20 定价是大规模获客策略。 OpenClaw 云托管面向的是更小但更高价值的人群——他们已经在花$80-360/月了。$40-80 是合理区间。

**2. 定价结构比价格点更重要**

固定订阅简单但粗暴，按量计费公平但焦虑，Vercel 模式优美但受众有限。大概率的最优解是混合：基础订阅+弹性额度。

**3. "不用操心"是核心卖点，但需要翻译**

不能直接说"你的时间值$50/h"——要用案例、用对比、用社区口碑让用户自己得出这个结论。

**4. BYOK vs 统一 Key 是阶段性决策**

早期 BYOK 建信任，规模化后引入统一 Key 提利润。混合模式是终局。

**5. AI SaaS 的毛利天然低于传统 SaaS**

因为边际成本不趋近于零。这意味着要么走量（薄利多销），要么走价值（高单价高服务），中间地带很难活。

## 第七章 · 商业模式可行性推演：开源产品怎么赚钱活下去

前六章拆完了成本结构和定价逻辑，现在回到那个最本质的问题： **OpenClaw 云托管，到底能不能赚钱？**

这不是一个拍脑袋的问题。开源商业化是一条铺满前辈尸体的路——有成功上市的（GitLab 、 MongoDB 、 Elastic），有被收购的（MySQL 、 Docker），也有估值暴跌的（Confluent 、 HashiCorp）。开源产品的商业化难度远高于纯 SaaS，原因很简单：你的核心代码是免费的，用户随时可以自己跑。

但难不代表不可能。关键在于找到正确的商业模式，以及在对的时间窗口执行。

让我们从四条经典路径开始推演。


### 开源商业化四条路径

#### 路径一：Open Core（GitLab 模式）

**核心逻辑：** 基础版开源免费，高级功能收费。

GitLab 是这条路的标杆——社区版免费，企业版按座位数收费，高级安全审计、合规工具、高级 CI/CD 功能锁在付费墙后面。 MongoDB 类似：Community Server 开源，Atlas 的自动分片、加密存储、高级监控是付费项。

**OpenClaw 适配度：中等。**

理论上可行的收费功能：
- 高级安全沙箱（microVM 隔离）
- 企业级审计日志
- 多租户管理控制台
- 高级记忆分析（语义热力图、用户画像自动提取）
- 优先技术支持

但问题在于 OpenClaw 的核心架构——Gateway 、 Agent 、 Memory 、 Skills——全部开源，且社区已经围绕这些核心能力建立了大量工具和工作流。你要在一个完全开放的代码库上划出"免费"和"收费"的边界线，这条线太靠左，用户不买单（"这个功能 GitHub 上有替代品"）；太靠右，开源社区会觉得被背叛（"你们在开源外壳里塞私有功能"）。

::: warning ⚡ PM 旁注
Open Core 最大的陷阱不是技术问题，是社区关系管理。 Elastic 被 AWS 用开源代码推出 OpenSearch 的教训还在眼前——许可证之争的本质是商业利益的重新分配。
:::

**结论：能做，但刀法要精准。适合作为辅助收入来源，不适合作为主营模式。**

#### 路径二：Hosted Service（Vercel 模式）

**核心逻辑：** 代码开源自部署免费，我来帮你托管就收费。

Vercel（托管 Next.js）、 Supabase（托管 PostgreSQL）、 PlanetScale（托管 MySQL/Vitess）走的都是这条路。用户完全可以自己部署——但大多数人不想，也没时间。你卖的不是代码，是 **省心**。

**OpenClaw 适配度：最高。**

这里不需要重复论证——Day 3 已经用血泪证明了自部署 OpenClaw 的门槛。 3-4 小时的首次部署、月均 4-9 小时的运维、 20-40 小时的学习曲线。这些是真实的痛点，不是营销话术。

更关键的是，OpenClaw 的架构天然适合托管化：
- Gateway 本身就是一个网络服务，从单机到云端的改造路径清晰
- 多渠道连接（23 个渠道插件的 token 刷新、 API 适配、断线重连）是最适合被平台化的运维负担
- Agent 的 workspace 可以映射为云端存储卷
- 记忆系统的向量搜索可以共享基础设施降低成本

Vercel 的增长曲线是最好的参照：Next.js 月下载量突破 2000 万，但绝大多数生产部署跑在 Vercel 上而非自建服务器。 **开发者用脚投票——他们选择了"不用操心"。**

**结论：主营模式的最佳候选。 Day 3 证明了需求真实存在，架构改造路径清晰，单位经济学可算。**

#### 路径三：Support & Consulting（Red Hat 模式）

**核心逻辑：** 软件免费，企业级支持和咨询收费。

Red Hat 靠 RHEL（Red Hat Enterprise Linux）的订阅式企业支持做到了 340 亿美元被 IBM 收购的结局。核心卖点不是 Linux 本身（CentOS/Ubuntu 都免费），而是企业用 Linux 时需要的那套保障体系——安全补丁 SLA 、合规认证、 7×24 技术支持、迁移咨询。

**OpenClaw 适配度：当前偏低，未来可能提升。**

原因直接：Support 模式需要足够大的企业用户基础。当前 OpenClaw 的用户画像以开发者和技术爱好者为主，企业客户——尤其是愿意为技术支持付费的大中型企业——数量未经验证。

但有一个值得关注的信号： **Agent 正在从开发者玩具变成企业工具。** 如果你相信"每家公司都会有自己的 AI Agent"这个趋势判断，那么企业对 Agent 基础设施的支持需求将急剧增长。 OpenClaw 作为 Agent 运行时的事实标准（280K Stars 、 1075+贡献者、 13700+ Skills），有可能成为企业 Agent 基础设施的首选——届时，支持和咨询的需求会自然涌现。

**结论：不适合冷启动阶段的主营模式。但当企业用户规模达到一定体量后，可以作为高利润的补充业务。**

#### 路径四：Marketplace（WordPress 模式）

**核心逻辑：** 建立技能/插件市场，从交易中抽成。

WordPress.org 的插件和主题市场、 Shopify 的 App Store 、 Salesforce 的 AppExchange——平台吃生态红利的经典模式。 OpenClaw 已经有了 ClawHub（13700+ Skills），生态骨架隐约可见。

**OpenClaw 适配度：长期高，短期低。**

短期挑战：
- Day 3 的生态分析显示，当前 ClawHub 的 Skill 质量参差不齐（约 20%存在安全或质量问题）
- 付费 Skill 的市场教育未完成——用户习惯了免费开源，你要让他们为一个 Skill 付费需要证明付费版确实比免费版好 10 倍
- 平台抽成的合理比例是一个敏感话题（苹果 30%被骂了多少年？）

长期机会：
- 如果 OpenClaw 成为 Agent 的事实运行时，"给 Agent 装一个新技能"就像"给手机装一个新 App"——这个类比的商业价值不用多说
- 企业级 Skill（金融数据接入、 CRM 集成、合规审计）有明确的付费意愿
- 平台可以通过认证、安全审计、性能保证等增值服务提升付费 Skill 的价值

**结论：当前不具备条件。需要生态繁荣度达到临界点——参考 WordPress 生态，至少需要数千个高质量付费插件和一个活跃的开发者经济。但如果 Day 3-5 描述的生态飞轮持续转动，3-5 年后这是一个千亿级的想象空间。**


### 单位经济学推算：一笔从零算到一的账

理论分析给了方向，但投资人和老板要看数字。让我们以 **Hosted Service（路径二）** 为主营模式，做一轮单位经济学推算。

#### 假设条件

| 参数 | 数值 | 依据 |
|------|------|------|
| 目标定价 | Pro 档 $40/月 | 第六章市场定价参照系（Supabase Pro $25，Vercel Pro $20，考虑 Agent 复杂度溢价） |
| 初始付费用户 | 1,000 人 | 保守假设，对标 Dify 早期 |
| BYOK 用户占比 | 60% | 开发者群体偏好自带 Key |
| 统一 Key 用户占比 | 40% | 便利性驱动 |

#### 收入侧

```
MRR（月经常性收入）= 1,000 × $40 = $40,000/月
ARR（年经常性收入）= $40,000 × 12 = $480,000/年
```

$48 万年收入，对于一个早期产品来说不算小。但我们接着往下算。

#### 成本侧

**每用户基础设施成本：$5-8/月**

拆解：
- 计算资源（容器/VM）：$2-4/月（多租户共享，非独占）
- 存储（workspace + 向量数据库）：$1-2/月
- 网络（入站渠道消息 + 出站 API 调用）：$0.5-1/月
- 监控、日志、备份：$0.5-1/月

这个区间基于 AWS/GCP 的多租户架构估算。如果用 Hetzner 等低成本提供商，还能进一步压缩。

**每用户 API 成本：差异巨大**

这里是 BYOK 策略最显著的影响：

| 用户类型 | 每用户 API 月成本 | 平台承担 |
|----------|---------------|---------|
| BYOK 用户（60%） | $0 | $0——用户自带 Key，API 成本与平台无关 |
| 统一 Key 用户（40%） | $15-25/月（用户账单） | $0——通过用量计费转嫁给用户 |

**等等，统一 Key 模式下平台真的零成本吗？**

不完全是。如果平台作为 API 中间商（批量采购 Key 、统一计费），存在几种成本结构：

1. **纯转嫁模式**：平台只做计费通道，API 成本 100%转嫁用户，平台加收 10-15%的"管理费"。毛利最高，但用户可能质疑"我为什么不自己买 Key"。
2. **打包模式**：Pro 档包含一定额度的 API 调用（比如每月$20 额度的 Claude 调用），超出部分按量计费。用户体验更好，但平台承担了 API 成本的波动风险。
3. **混合模式**：基础款 BYOK，高级款打包。灵活但复杂。

为了简化计算，假设 **打包模式**，每个统一 Key 用户平台平均承担$15/月的 API 成本：

```
统一Key用户API成本 = 400 × $15 = $6,000/月
```

#### 毛利推算

| 项目 | 金额/月 |
|------|---------|
| **总收入**| $40,000 |
| 基础设施成本（1000 用户 × $6.5 平均） | -$6,500 |
| API 成本（400 统一 Key 用户 × $15） | -$6,000 |
| **毛利**| **$27,500**|
| **毛利率**| **68.75%**|

如果是纯 BYOK 模式（假设所有用户自带 Key）：

```
毛利 = $40,000 - $6,500 = $33,500
毛利率 = 83.75%
```

如果是纯统一 Key 模式（假设所有用户用平台 Key）：

```
毛利 = $40,000 - $6,500 - $15,000 = $18,500
毛利率 = 46.25%
```

**一张表说清楚三种模式的毛利差异：**

| Key 策略 | 毛利率 | 特征 |
|---------|-------|------|
| 纯 BYOK | ~84% | 高毛利，但用户门槛高（需自行获取 API Key） |
| 混合（60/40） | ~69% | 均衡，适合多元用户群 |
| 纯统一 Key | ~46% | 低毛利，但用户体验最顺滑 |

::: warning ⚡ PM 旁注
毛利率 46%在 SaaS 行业偏低（优秀 SaaS 毛利通常>70%）。这就是为什么 **BYOK 不是可选项，而是毛利的命门**。你必须想办法让至少一半以上的用户自带 Key 。
:::


### CAC vs LTV：获客成本与用户终身价值

有了毛利，我们还需要知道： **获取一个用户要花多少钱？这个用户能给你赚回多少钱？**

#### CAC（客户获取成本）

开源产品在获客上有一个结构性优势： **社区即渠道。**

传统 SaaS 的获客路径：广告投放→落地页→试用→付费。每一步都有漏损，平均 CAC 在$200-500 之间（B2B SaaS 更高）。

开源产品的获客路径：GitHub 发现→自部署试用→遇到运维痛点→转向云托管。中间几乎没有广告成本，用户在使用开源版的过程中已经完成了产品教育和需求验证。

**估算 OpenClaw 云托管的 CAC：$50-100 。**

构成：
- 内容营销分摊（社区运营、技术博客、文档维护）：~$20
- 自然搜索/社区口碑带来的免费流量：$0
- 少量付费渠道（KOL 合作、技术社区赞助）：~$20-50
- 转化漏斗的产品成本（免费试用额度等）：~$10-30

与同类对标：
- Supabase CAC 估算：~$100-200
- Vercel CAC 估算：~$50-150（开发者口碑极强，付费获客比例低）
- 传统 B2B SaaS：$200-500

OpenClaw 的 280K Stars 和 10 万+中国活跃用户意味着一个庞大的"已教育"用户池。这些人已经知道产品是什么、能做什么——你只需要在他们遇到痛点时给出一个"点击即用"的云托管选项。获客成本天然偏低。

#### LTV（客户终身价值）

```
LTV = ARPU × 平均留存月数 × 毛利率
```

假设：
- ARPU（每用户平均收入）：$40/月
- 平均留存：12 个月（保守假设——Agent 产品的留存取决于用户的记忆积累，Day 2 已论证记忆带来的高切换成本）
- 毛利率：69%（混合模式）

```
LTV = $40 × 12 × 0.69 = $331
```

如果留存延长到 24 个月（乐观假设——用户记忆积累越多，切换成本越高）：

```
LTV = $40 × 24 × 0.69 = $662
```

#### LTV/CAC 比率

| 场景 | CAC | LTV | LTV/CAC | 判断 |
|------|-----|-----|---------|------|
| 保守（留存 12 月，CAC $100） | $100 | $331 | 3.3x | 及格 |
| 中性（留存 12 月，CAC $50） | $50 | $331 | 6.6x | 健康 |
| 乐观（留存 24 月，CAC $50） | $50 | $662 | 13.2x | 优秀 |

SaaS 行业的健康基准：LTV/CAC > 3x 。上表显示即使在保守假设下，这个比率也刚好及格。在中性和乐观假设下，数字非常健康。

**但有一个关键的不确定性：留存。**

Agent 产品的留存曲线目前没有足够多的行业数据作为参考。 Day 1 提过工具类产品的 Day 30 留存常跌破 5%。但 OpenClaw 的记忆系统（Day 2 详述）理论上创造了强大的信息粘性——用户在 MEMORY.md 里投入的时间越长，切换到其他平台的心理成本越高。

这是一个需要验证的假设，不是已知的结论。 12 个月的平均留存是一个中间值猜测，实际可能更高（记忆粘性生效），也可能更低（Agent 工具的"尝鲜→受挫→流失"曲线）。


### Token 降价：是洪水还是涨潮？

第二章分析了 Token 定价的年均降幅（50-70%）。这个行业级的结构性趋势对商业模式有什么影响？

**场景一：统一 Key 模式下，Token 降价=利润提升**

如果平台以固定价格打包 API 调用额度（比如 Pro 档$40/月包含$20 的 API 额度），Token 降价意味着同样的$20 可以买到更多的调用次数。用户体验提升（配额更大方了），平台成本下降（同等用量花得更少了），但收入不变（还是$40/月）。

效果： **直接利好。** 毛利率会从 46%逐步攀升。

假设 Token 价格在两年内降 70%：
```
当前API成本（统一Key用户均）：$15/月
两年后API成本：$15 × 0.3 = $4.5/月
毛利率变化：从46% → 69%（几乎追平BYOK模式）
```

**场景二：BYOK 模式下，Token 降价对平台利润无直接影响**

用户自带 Key，API 费用与平台无关。 Token 降价$0.01 还是$0.001，平台的收入和成本都不变。

但有间接效应：Token 降价→使用 Agent 的经济门槛更低→更多人愿意用 Agent→Agent 市场总盘子变大→OpenClaw 用户基数扩大→更多人从自部署转向云托管。

效果： **间接利好。** 不影响单位经济学，但扩大了市场规模。

**场景三：所有模式下，Token 降价改变成本结构的权重**

这是最值得 PM 关注的长期趋势。

当前 OpenClaw 云托管的成本构成（混合 Key 模式）：
```
基础设施：52%（$6,500/$12,500）
API成本：48%（$6,000/$12,500）
```

Token 价格降 70%后：
```
基础设施：78%（$6,500/$8,300）
API成本：22%（$1,800/$8,300）
```

**成本结构的重心从"API 调用"转向"基础设施"。** 这意味着：
1. 降低 API 成本的优化策略（模型选择、 Prompt 精简）的杠杆在减小
2. 降低基础设施成本的优化策略（多租户、资源复用、低成本服务商）的杠杆在增大
3. 规模效应变得更重要——基础设施成本有明显的规模递减特征，API 成本没有

::: warning ⚡ PM 旁注
**Token 降价是 Agent 行业的"摩尔定律"。** 就像芯片降价推动了 PC 普及、带宽降价推动了视频流媒体，Token 降价正在推动 AI Agent 从极客玩具走向大众工具。顺势而为的产品会享受到行业的结构性红利。
:::


### ☁️ 观测点③：规模效应拐点——什么时候能赚钱？

推演了收入、成本、毛利、获客之后，回到投资人和老板最关心的问题： **什么时候能赚钱？**

这取决于固定成本（团队薪资、基础设施底线、运营开支）与边际收入的交叉点。

#### 固定成本估算

一个最小可行的云托管团队（MVP 阶段）：

| 项目 | 月成本 | 说明 |
|------|--------|------|
| 工程团队（5 人） | $60,000 | 2 后端+1 前端+1SRE+1 安全，按湾区中位数 |
| 产品+设计（2 人） | $25,000 | 1PM+1 设计 |
| 基础设施底线 | $5,000 | 即使零用户也要维持的服务器、监控、 CI/CD |
| 运营杂项 | $5,000 | 法务、合规、工具订阅等 |
| **合计**| **~$95,000/月**| |

如果在中国组建团队，固定成本可以压到$40,000-50,000/月。以下按$50,000/月计算（中国团队假设）。

#### 盈亏平衡推算

```
盈亏平衡用户数 = 固定成本 / 每用户毛利
每用户毛利 = $40 × 69% = $27.6/月（混合Key模式）
盈亏平衡用户数 = $50,000 / $27.6 ≈ 1,812 人
```

**约 1,800 个付费用户时达到盈亏平衡。**

#### 不同规模阶段的利润图景

| 付费用户数 | MRR | 总毛利/月 | 固定成本/月 | 净利/月 | 状态 |
|-----------|-----|----------|-----------|--------|------|
| 500 | $20,000 | $13,800 | $50,000 | **-$36,200**| 烧钱 |
| 1,000 | $40,000 | $27,600 | $50,000 | **-$22,400**| 烧钱 |
| 1,800 | $72,000 | $49,680 | $50,000 | **≈$0**| 盈亏平衡 |
| 3,000 | $120,000 | $82,800 | $55,000 | **+$27,800**| 盈利 |
| 5,000 | $200,000 | $138,000 | $60,000 | **+$78,000**| 规模效应显现 |
| 10,000 | $400,000 | $276,000 | $80,000 | **+$196,000**| 利润飞轮 |

注意固定成本在 5,000-10,000 用户阶段也会增长（需要更大团队、更多基础设施），但增速远低于收入增速——这就是 **规模效应（Economies of Scale）**。

几个关键节点：
- **1,000 用户以下**：固定成本吃掉大部分甚至全部收入。这是纯烧钱阶段，需要外部资金支持
- **1,000-1,800 用户**：仍在亏损，但亏损在收窄。产品-市场匹配（PMF）的关键验证窗口
- **1,800-5,000 用户**：跨过盈亏平衡后加速盈利。每新增一个用户，边际利润几乎等于毛利
- **5,000 用户以上**：规模效应明显。基础设施的多租户共享、 API 的批量折扣、品牌口碑带来的低 CAC，三者叠加形成利润飞轮

#### 关键问题：付费转化率

上面所有推算都建立在"能获得 X 个付费用户"的假设上。但开源产品的付费转化率一直是行业之谜。

参考数据：
- **Dify**（开源 LLM 应用开发平台）：估算付费转化率 2-5%
- **GitLab**：早期社区版到企业版的转化率约 1-3%
- **Supabase**：免费到付费的转化率估算 5-8%（开发者工具中偏高）
- **WordPress.com**（注意不是 self-hosted WordPress）：约 3-5%

如果 OpenClaw 有 10 万+中国活跃用户（Day 1 引用数据），假设其中有一定比例有云托管需求：

| 转化率 | 付费用户数 | 够不够盈亏平衡？ |
|--------|-----------|-----------------|
| 1% | 1,000 | ❌ 不够 |
| 2% | 2,000 | ✅ 刚过 |
| 3% | 3,000 | ✅ 开始盈利 |
| 5% | 5,000 | ✅ 规模效应 |

**2%的付费转化率是生死线。** 低于 2%，需要更多外部资金或更低的固定成本才能存活。高于 3%，商业模式基本跑通。

但要注意"10 万活跃用户"这个数据的来源未经第三方验证（Day 1 已讨论），且"活跃用户"到"有云托管需求的用户"之间还有一层筛选。实际可触达的转化池可能比 10 万小得多。

::: warning ⚡ PM 旁注
**不要高估转化率，不要低估市场增长。** 开源 Agent 产品是一个增量市场——目前全球 Agent 用户总量还在快速膨胀，OpenClaw 的用户基数本身也在增长。即使转化率只有 1%，如果用户基数从 10 万增长到 100 万，付费用户也能到 1 万。增长比转化率更重要。
:::


## 第八章 · 算完账之后：三个判断标准

七章看下来，数字多、变量多、假设多。如果你是一个需要做决策的人——无论是决定"要不要做云托管"还是决定"我个人要不要用云托管"——你需要把复杂的分析简化为几个清晰的判断标准。

这是本篇最后的收束，也是 Day 6 要传递的核心决策框架。

### 标准一：你的时间值多少钱？

这是最容易被忽略、又最有杀伤力的一个标准。

Day 3 我们算过自部署 OpenClaw 的时间成本：
- 首次部署：3-4 小时
- 月均运维：4-9 小时
- 学习曲线：20-40 小时（首年摊销约 2 小时/月）
- **月均时间投入：6-13 小时**

如果你的时间价值超过$30/小时（这是一个美国初级开发者的时薪下限），那么：

```
自部署时间成本 = 6-13小时 × $30 = $180-390/月
云托管费用 = $40/月

差额 = $140-350/月
```

**云托管每月帮你省下$140-350 的时间成本。**

但如果你是学生、 hobby 开发者、或者时间不值钱的退休工程师呢？

```
自部署时间成本 = 6-13小时 × $10 = $60-130/月
云托管费用 = $40/月

差额 = $20-90/月
```

省钱幅度缩小，但仍然为正。

**转折点在时薪约$5 以下** ——当你的时间成本极低时，自部署的总成本（$12 服务器+$60-130 时间成本=$72-142/月）反而可能低于云托管+API 费用的总包。但这类用户不是云托管的目标客户。

::: warning ⚡ PM 旁注
人们天生低估自己的时间成本。"我自己弄也就几个小时"——问题是这几个小时每个月都要重复，而且 debug 的时间永远比预期多。
:::

**判断公式：** 
- 时薪 > $30 → 云托管几乎一定更划算
- 时薪 $10-30 → 需要具体情况具体分析
- 时薪 < $10 → 自部署可能更经济

### 标准二：你需要多少控制权？

这不是一个成本问题，是一个价值观问题。

自部署 OpenClaw 给你的控制权：
- **数据主权**：所有数据（聊天记录、 MEMORY.md 、 SOUL.md）都在你的服务器上
- **架构控制**：可以自定义 Gateway 配置、修改源码、添加私有 Skill
- **审计能力**：完全可见的系统日志，不经过任何第三方
- **离线能力**：不依赖任何云服务商的可用性

云托管需要你让渡的控制权：
- 数据经过平台服务器（即使加密传输，平台理论上有访问能力）
- 架构由平台决定（多租户隔离程度、资源分配）
- 审计依赖平台提供的日志接口（完整性取决于平台）
- 平台宕机=你的 Agent 停摆

**对控制权需求最强的三类用户：**

1. **金融/医疗/法律行业**：监管要求数据不能出境、不能经过第三方，自部署几乎是唯一选项
2. **安全敏感型极客**："我的数据只经过我的服务器"不是 paranoia（偏执），是原则
3. **深度定制用户**：需要魔改源码、集成私有系统的企业

**对控制权需求最弱的三类用户：**

1. **个人开发者/创客**："能用就行，别让我配环境"
2. **小团队/初创公司**："我们没有 SRE，也养不起"
3. **非技术用户**："Docker 是什么？" ——这类用户甚至不知道自己在让渡什么

**判断公式：**
- 数据主权是刚需（行业监管/企业安全策略） → 自部署
- 数据主权是偏好但非刚需 → 看平台的安全承诺和审计机制
- 数据主权不在考虑范围 → 云托管

### 标准三：你的使用规模有多大？

个人使用和团队/企业使用的经济学完全不同。

**个人用户（1 个 Agent，1-3 个渠道）：**
- 自部署成本：$12/月（VPS）+ 时间
- 云托管成本：$40/月
- 规模效应：无。你只有一个 Agent，无法从多租户共享中受益

这种场景下，自部署的硬成本确实更低（如果你不计算时间）。云托管的溢价主要来自"省时间"和"省操心"。

**团队用户（3-10 个 Agent，5-10 个渠道）：**
- 自部署成本：$50-100/月（更强的服务器）+ 专人运维时间
- 云托管成本：$100-200/月（按 Agent 数计费）
- 规模效应：开始显现。平台的多 Agent 管理、统一监控、团队协作功能节省的运维时间显著

这种场景下，云托管的 ROI（投资回报率）明显提升。不是因为单价便宜了，而是因为 **运维复杂度随 Agent 数量非线性增长** ——Day 2 第七章已经详细分析过，5 个 Agent×3 个渠道的运维成本不是×15 而是×30 。云托管把这个非线性增长拉平。

**企业用户（10+个 Agent，全渠道覆盖）：**
- 自部署成本：$200-500/月 + 至少半个 SRE 的人力
- 云托管成本：Enterprise 档$500-1000/月
- 规模效应：最大化。统一管理、合规审计、 SLA 保障、优先支持——每一项都是自建时需要大量人力的项目

**判断公式：**
- 1 个 Agent → 自部署可能更经济（如果你技术过关且时间充裕）
- 3-10 个 Agent → 云托管的 ROI 显著
- 10+个 Agent → 云托管几乎是必选（除非你有专职运维团队）


### PM 核心收获

写了将近两万字（Day 6 全文），五个核心要点带走：

**① Token 经济学是 Agent 产品的命脉。**
不理解 Token 定价机制，就无法理解 Agent 产品的成本结构、定价逻辑、甚至用户行为。模型选择是 10x 的成本杠杆——同一个 Agent 任务，用 Claude Opus 和用 Claude Haiku 的成本差了 18 倍。 PM 不需要会调模型，但必须理解这个杠杆的存在。

**② 隐性成本远超显性成本。**
服务器每月$12，看起来很便宜。但加上时间成本（$150-450/月）、学习成本（首年$200-400/月摊销），真实 TCO 是显性成本的 5-10 倍。这不是 OpenClaw 的问题，这是所有 self-hosted 产品的共同特征——只是在 Agent 领域被放大了。

**③ 云托管卖的不是功能，是"不用操心"。**
功能上，云托管和自部署跑的是同一套代码。用户为什么付费？为的是不用半夜修 webhook 、不用追 Token 账单、不用 debug 渠道连接。"省心"是一种真实的产品价值，尤其对时间成本高的用户。

**④ BYOK 是毛利的命门。**
BYOK 模式下毛利率可达 84%，纯统一 Key 模式只有 46%。这不是一个"可选的功能"，而是决定商业模式健康度的核心策略。好的产品设计应该让 BYOK 成为默认推荐项，同时为不想折腾的用户提供统一 Key 的便利。

**⑤ Token 降价是行业结构性利好。**
Token 价格每年降 50-70%，无论 BYOK 还是统一 Key，整个 Agent 市场都在受益。对平台来说，统一 Key 模式下 Token 降价直接提升毛利；对生态来说，Token 降价降低了用 Agent 的门槛，扩大了市场。这是 AI Agent 行业的"摩尔定律"，顺势而为者受益。


### 深度思考题

**Q1：Token 价格再降 90%后，OpenClaw 的成本结构会怎么变？**

提示：当 API 调用的成本趋近于零，Agent 产品的成本中心从"调用模型"转向"运行基础设施"和"维护平台"。这时候，产品的竞争从"谁能更便宜地调用模型"变成"谁能提供更好的基础设施体验"。想想 Vercel——它卖的不是 Node.js 运行时（那是免费的），而是部署体验和边缘网络。 OpenClaw 的未来也可能类似：卖的不是模型调用（那会很便宜），而是 Agent 管理和编排体验。

**Q2：如果你是 OpenClaw 云托管的 PM，你会选 BYOK 还是统一 Key？**

提示：这不是二选一的问题，而是一个用户分群的问题。开发者和技术用户偏好 BYOK（控制感+成本透明）；非技术用户偏好统一 Key（简单+不用管）；企业用户可能偏好统一 Key（合规审计更方便——一个供应商账单比员工各自的 API Key 账单好管理得多）。好的答案不是"选哪个"，而是"怎么让不同用户自然流向适合他们的方案"。

**Q3：企业客户说"成本不是问题，安全才是"——这句话的隐含假设是什么？**

提示：这句话至少包含三个隐含假设：
1. 企业有足够的预算来为安全溢价买单（成本弹性高）
2. 安全是一个可以用钱解决的问题（如果你提供足够好的安全方案，他们愿意付费）
3. **当前市场上没有既安全又便宜的方案。** 一旦 Token 降价到一定程度，安全方案的成本会大幅下降，"成本不是问题"这句话的前提可能就不成立了——因为安全不再需要溢价。
4. 更深层的假设：说这句话的人往往不是直接付钱的人。当 CTO 说"成本不是问题"时，CFO 可能有不同看法。


### Day 7 预告

## 附录：术语表

| 术语 | 英文全称 | 中文解释 |
|------|---------|---------|
| **TCO**| Total Cost of Ownership | 总拥有成本。包括直接购买/订阅费用、运维费用、时间成本等所有与使用某产品/服务相关的成本总和 |
| **Token**| — | AI 模型处理的最小文本单位。大约 1 个 Token≈4 个英文字符≈0.75 个汉字。模型按 Token 数量计费 |
| **BYOK**| Bring Your Own Key | 自带密钥。用户自行向模型供应商（如 Anthropic 、 OpenAI）购买 API Key，平台只做转发不做计费 |
| **MRR**| Monthly Recurring Revenue | 月经常性收入。 SaaS 产品的核心财务指标，衡量每月可预期的稳定收入 |
| **CAC**| Customer Acquisition Cost | 客户获取成本。获得一个新付费客户平均需要花费的总金额 |
| **LTV**| Lifetime Value | 客户终身价值。一个客户在整个使用生命周期内为产品带来的总毛利 |
| **毛利率**| Gross Margin | （收入-直接成本）÷ 收入 × 100%。衡量每一元收入扣除直接成本后的盈利能力。 SaaS 行业健康毛利率通常>70% |
| **规模效应**| Economies of Scale | 随着用户/产量增加，单位成本递减的经济现象。云托管的基础设施成本具有明显的规模效应——1 万用户的人均基础设施成本远低于 100 用户 |
